# RadiProtocol — Agent Instructions

**Development workflow:** see [`docs/DEVELOPMENT-WORKFLOW.md`](docs/DEVELOPMENT-WORKFLOW.md) for the authoritative process. Keep this file short and current: it is loaded into agent sessions.

## Project snapshot

- RadiProtocol is an Obsidian plugin for guided radiology protocol execution.
- Source is TypeScript in `src/`, bundled by esbuild into root `main.js`.
- CSS source lives in `src/styles/`; root `styles.css` is generated by esbuild and must not be edited directly.
- `.rp.json` is the target protocol document format. Legacy `.canvas` compatibility may still exist in tests/migration paths; do not expand it unless explicitly scoped.
- Web/backend/auth/payment rewrites are out of scope unless the task explicitly says otherwise.

## Code map

- `src/main.ts` — plugin registration, commands, settings wiring, inline-runner registry.
- `src/protocol/` — `.rp.json` document model, parser, store, write coordination.
- `src/graph/` — protocol graph types/validation and compatibility helpers.
- `src/runner/` — pure protocol execution engine and runner state helpers.
- `src/views/` — Obsidian UI hosts/modals, including inline runner and protocol editor.
- `src/snippets/` — snippet discovery, binding, insertion, placeholder handling.
- `src/i18n/` — translation service and locale dictionaries.
- `src/constants/` — shared constants for CSS classes, runner/editor states, etc.
- `src/styles/` — feature-scoped CSS source files concatenated by `esbuild.config.mjs`.
- `src/__tests__/` and colocated `*.test.ts` files — Vitest coverage.
- `docs/adr/` — durable architecture decisions. Use ADRs for removals or direction changes.
- `.planning/` — local GSD scratchpad (gitignored; see DEVELOPMENT-WORKFLOW.md for policy). Git/GitHub/code/tests are authoritative, not planning docs.
- `docs/ARCHITECTURE-NOTES.md` — implementation-relevant notes promoted from planning.

## Authoritative commands

Use targeted checks while iterating, then the full gate before PR/release.

```bash
npm run build              # typecheck + production bundle; regenerates main.js/styles.css
npm run lint               # ESLint + Stylelint
npm test                   # Vitest suite
npm run check:planning     # local planning freshness guard
npm run check:consistency  # package/manifest/docs consistency
npm run check              # full gate: build + lint + tests + planning + consistency
```

For session start or release work, verify actual repo state before trusting notes:

```bash
git fetch --all --tags --prune
git status --short --branch
node -e "for (const f of ['package.json','manifest.json']) console.log(f, require('./'+f).version)"
git tag --sort=-v:refname | head -10
```

## Current runner architecture

- Protocol execution is **inline-only** as of RadiProtocol 1.15.0.
- Runtime host: `src/views/inline-runner-modal.ts`.
- Pure engine: `src/runner/protocol-runner.ts`.
- Plugin-level inline runner registry lives in `src/main.ts` and keys instances by `canvasPath#notePath`.
- Do **not** restore sidebar/tab `RunnerView` or deleted runner-view CSS to fix inline-runner issues. The decision is recorded in `docs/adr/0001-inline-runner-only.md`.
- Keep product session persistence separate from internal runner snapshots: do not delete snapshot helpers just because persisted session recovery is out of scope.

## CSS architecture

CSS is split into per-feature files under `src/styles/`. The build concatenates them in order from `CSS_FILES` in `esbuild.config.mjs`.

Current source files:

- `_utilities.css` — shared utilities and reset-level helpers.
- `protocol-editor.css` — independent `.rp.json` visual protocol editor.
- `snippet-manager.css` — snippet manager + placeholder chips.
- `snippet-fill-modal.css` — snippet fill-in modal.
- `loop-support.css` — loop iteration UI hooks shared by runner UI.
- `snippet-tree-picker.css` — snippet tree picker.
- `inline-runner.css` — inline runner modal UI.
- `donate-section.css` — donate/settings section.

When adding CSS for a new feature:

- create a new file in `src/styles/` and add it to `CSS_FILES` in `esbuild.config.mjs`, or edit only the relevant existing feature file;
- do not add CSS to an unrelated feature file;
- after any CSS change, run `npm run build` so generated `styles.css` is refreshed.

## Critical editing rules

### Never remove existing code you didn't add

When editing accumulated files such as `src/styles/*`, `src/main.ts`, `src/views/*`, or shared helpers:

- only add or modify code relevant to the current scoped task;
- never delete rules, functions, event listeners, i18n keys, tests, or fixtures that you did not prove are obsolete for this task;
- if unsure whether something belongs to a previous phase, leave it alone and report it.

Cleanup phases are the exception: if the explicit scope is to remove orphaned code/CSS, delete only references proven unused and verify with build/lint/tests.

### DOM/CSS restructuring pitfall

After moving elements between containers, grep for old container class names in CSS. Selectors scoped to the old parent silently stop matching and can cause subtle button sizing/layout regressions.

### Obsidian icon tooltip pitfall

When using Obsidian `setIcon(el, ...)`, do not set `title` on the same element. Use `aria-label` plus translated labels instead; `title` causes duplicate native/CSS hover tooltips.

### Runner layout pitfall

Inline runner modal layout must avoid divider/button jitter. Keep text/actions/footer zone behavior intentional, use stable min-heights where needed, and verify question/answer/snippet/loop states after layout changes.

## i18n rules

- User-facing strings must go through the translation service (`i18n.t`, `t`, or a passed `Translator`).
- Add/update both English and Russian dictionaries together.
- Check less obvious strings too: settings labels, dropdown options, headings, placeholders, modal buttons, aria labels, and empty-state text.
- Avoid hardcoded Russian/English UI text in new code unless it is a test fixture or non-user-facing diagnostic.

## Release/version rules

- Use `npm version X.Y.Z` for releases. The repository `.npmrc` sets `tag-version-prefix=""`, so tags are numeric like `1.17.12`, not `v1.17.12`.
- Do **not** run `node version-bump.mjs` directly. It relies on npm lifecycle environment and can corrupt version surfaces when run standalone.
- Keep `package.json`, `manifest.json`, `versions.json`, README latest-release text, git tag, and GitHub Release aligned.
- For BRAT/GitHub releases, verify the actual GitHub Release assets (`main.js`, `manifest.json`, `styles.css`), not just local files or tags.

## Agent execution protocol

For external coding agents or subagents:

- Keep tasks bounded: one feature/fix slice, explicit allowed files, explicit forbidden actions.
- Forbid by default: git commit, git push, force-push, deleting unrelated files, production/release config changes.
- Prefer read-only analysis before large refactors, then make mechanical edits with exact diffs.
- Agent self-reports are not proof. Verify with `git diff --stat`, inspect key diffs, and run targeted/full checks.
- If an agent leaves partial changes, audit the diff before rerunning or reverting; small test/assertion fixes are often faster by hand.
