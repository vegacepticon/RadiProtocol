# v1.4 Learnings: Snippets and Colors

## Decisions

### D-01: Centralize color injection inside saveNodeEdits, not at call sites

Color injection was placed inside `saveNodeEdits` before the Pattern B / Strategy A fork, so callers never need to pass `color` explicitly. This eliminated the root cause of color drift: previously, color was only written when `onTypeDropdownChange` fired, meaning field-only saves (question text, etc.) never re-injected the correct color.

**Rationale:** Single injection point serves both save paths (live canvas API and vault.modify) with one code change. Callers remain unchanged.

**Source:** 28-CONTEXT.md (D-01, D-02), 28-RESEARCH.md

### D-02: enrichedEdits spread copy before forking into save paths

Created `const enrichedEdits = { ...edits }` at the top of `saveNodeEdits` to inject color without mutating the caller's object. All downstream code (saveLive, Strategy A write loop) uses enrichedEdits instead of edits.

**Rationale:** Avoids side-effects on caller data while keeping the injection before the Pattern B / Strategy A fork.

**Source:** 28-01-PLAN.md, 28-01-SUMMARY.md

### D-03: Snippet node gets purple (palette "6") â€” the only unused color slot

All 6 Obsidian palette colors were already assigned to the 7 existing node kinds. Purple ("6") was the only unused slot, making the decision trivial.

**Rationale:** No color sharing needed; snippet nodes are visually distinct from all other kinds.

**Source:** 29-CONTEXT.md (D-11), 29-DISCUSSION-LOG.md

### D-04: subfolderPath re-derived from graph at getState() time, not persisted in runner state

The runner does not store `subfolderPath` as a field. Instead, `getState()` reads it from `graph.nodes.get(currentNodeId)` on demand. This keeps the session payload unchanged and avoids duplicate state.

**Rationale:** The subfolderPath is already in the graph model (parsed from canvas). Storing it again in runner state would create a sync hazard.

**Source:** 30-02-SUMMARY.md, 30-CONTEXT.md (D-22)

### D-05: New awaiting-snippet-pick RunnerStatus instead of reusing at-node

A dedicated runner status was added for the snippet picker state, rather than overloading the existing `at-node` status. This keeps the discriminated union clean and allows RunnerView to render a purpose-built picker UI.

**Rationale:** Reusing `at-node` would have required runtime checks to distinguish "normal at-node" from "snippet picker open," making the code fragile and harder to test.

**Source:** 30-CONTEXT.md (D-06), 30-DISCUSSION-LOG.md (Q6)

### D-06: Path-safety gate rejects traversal tokens outright, not normalizes them away

`SnippetService.listFolder` rejects any path containing `..` or `.` segments before any disk I/O, rather than normalizing them and checking containment. This is stricter than the plan originally sketched.

**Rationale:** Normalizing `../../etc` silently collapses to `etc` and passes containment checks â€” the opposite of the intended security posture. Rejecting early is both simpler and safer.

**Source:** 30-01-SUMMARY.md (deviation auto-fix), 30-CONTEXT.md (D-20)

### D-07: UndoEntry.returnToBranchList flag for branch-entered snippet picker step-back

When a user enters a snippet picker via a question branch (Phase 31), the undo entry is tagged with `returnToBranchList: true`. Step-back sees this flag and returns to the question's branch list rather than to the question's predecessor.

**Rationale:** Without the flag, step-back from a branch-entered picker would skip past the question entirely, losing the ability to try a different branch.

**Source:** 31-CONTEXT.md (D-08), 31-03-SUMMARY.md

### D-08: Validator message made branch-agnostic ("has no outgoing branches")

The graph validator's dead-end question message was reworded from "has no answers" to "has no outgoing branches" to accommodate mixed answer+snippet branching without adding new validation rules.

**Rationale:** The validation rule itself (outgoing.length === 0) is correct for both answer-only and mixed questions. Only the human-readable message needed updating.

**Source:** 31-CONTEXT.md (D-07), 31-01-SUMMARY.md

### D-09: Per-node snippet separator uses a different field name than answer separator

`SnippetNode` uses `radiprotocol_snippetSeparator` instead of reusing `radiprotocol_separator` from AnswerNode. This allows `resolveSeparator` to branch on `node.kind` and read the correct field.

**Rationale:** SnippetNode's separator has distinct semantics (controls join between snippet text and accumulated text, not between answer text and prior text). A separate field name prevents accidental cross-kind coupling.

**Source:** 31-CONTEXT.md (D-04), 31-PATTERNS.md (section 4d)

### D-10: Drill-down picker navigation is view-local, not persisted in session

The breadcrumb depth in the snippet picker is not serialized. On session resume, the picker resets to the snippet node's configured root. The user re-navigates if needed.

**Rationale:** Serializing drill-down state adds complexity disproportionate to the UX benefit (users rarely resume mid-drill-down).

**Source:** 30-CONTEXT.md (D-23), 30-03-SUMMARY.md

## Lessons

### L-01: Exhaustive switch/Record<RPNodeKind, string> catches all downstream consumers automatically

Adding `'snippet'` to `RPNodeKind` surfaced compile errors in every file that used exhaustive switches or `Record<RPNodeKind, ...>`: NODE_COLOR_MAP, canvas-parser, graph-validator, protocol-runner. This made it impossible to forget an integration point.

**When to apply:** Any time a discriminated union is extended, rely on TypeScript exhaustiveness to find all consumers rather than manually auditing.

**Source:** 29-01-SUMMARY.md (auto-fixed ProtocolRunner exhaustive switch)

### L-02: TDD Wave 0 (RED tests first) catches plan sketch bugs before implementation

In Phase 28, the RED test phase established the exact contract before any production code was written. In Phase 30, the TDD tests for `listFolder` caught a bug in the plan's path-normalization sketch: the plan suggested filtering out `..` segments, but the test correctly expected rejection.

**When to apply:** Complex logic (especially security-sensitive paths) benefits from writing tests first â€” the test serves as a correctness oracle for the plan itself.

**Source:** 28-00-SUMMARY.md, 30-01-SUMMARY.md (deviation: plan's path-normalization was buggy)

### L-03: Fixtures need realistic preludes for meaningful undo/step-back tests

Phase 30's initially-sketched fixture (`start -> snippet`) left the undoStack empty when the runner reached the snippet node, making step-back tests impossible. Adding a question+answer prelude pushed a real undo entry, enabling meaningful step-back verification.

**When to apply:** Whenever testing undo/step-back behavior, the fixture must include at least one user-driven choice before the target node.

**Source:** 30-02-SUMMARY.md (decision 1, deviation 2)

### L-04: Code review findings may reference non-existent field names

Phase 30's WR-02 review finding described `awaitingSnippetPick`, `pendingSnippetNodeId`, and `snippetCandidates` â€” field names that do not exist in the codebase. The fixer correctly skipped the finding after verifying the actual implementation already handles the described scenario. Review findings should always be validated against current source before applying fixes.

**When to apply:** When fixing code review items, always grep for the referenced identifiers first. AI-generated reviews can hallucinate field names.

**Source:** 30-REVIEW-FIX.md (WR-02 skipped)

### L-05: Void IIFE pattern enables async work inside synchronous buildKindForm

The EditorPanel's `buildKindForm` is synchronous, but populating the subfolder dropdown requires async vault listing. The `void (async () => { ... })()` pattern was used to launch async work without refactoring the synchronous method signature.

**When to apply:** When a synchronous Obsidian lifecycle method needs to trigger async I/O for UI population.

**Source:** 29-02-SUMMARY.md, 29-PATTERNS.md (section 7)

### L-06: Removing duplicate logic from onTypeDropdownChange after centralizing in saveNodeEdits

After Phase 28 centralized color injection in `saveNodeEdits`, the duplicate color injection in `onTypeDropdownChange` became a maintenance hazard (WR-02). Removing it required verifying that no test explicitly expected `onTypeDropdownChange` to inject color.

**When to apply:** After centralizing logic, actively search for and remove duplicates in callers to prevent divergence.

**Source:** 28-REVIEW.md (WR-02), 28-REVIEW-FIX.md

### L-07: BFS without visited set can hang on NTFS junction points

The initial `listSnippetSubfolders` BFS implementation had no cycle detection. On Windows (the target platform), NTFS junction points can create directory cycles that cause infinite traversal.

**When to apply:** Any filesystem BFS/DFS on Windows must include a visited set.

**Source:** 29-REVIEW.md (WR-01), 29-REVIEW-FIX.md

### L-08: State-shape widening cascades through 6+ files in discriminated union architectures

Adding `'awaiting-snippet-pick'` to `RunnerStatus` required synchronized changes in: runner-state.ts (type), protocol-runner.ts (getState, getSerializableState, restoreFrom), session-model.ts (PersistedSession), and runner-view.ts (render switch). Missing any one causes a TypeScript error.

**When to apply:** When planning a new runner status, budget for the full cascade: status union, state interface, getState case, serialization, deserialization, persisted model, and view switch.

**Source:** 30-02-SUMMARY.md (patterns-established, deviation 1)

## Patterns

### P-01: Undo-before-mutate invariant for user-action methods

Every runner method that mutates state (chooseAnswer, pickSnippet, chooseSnippetBranch) must push an UndoEntry with a spread copy of `loopContextStack` BEFORE any assignment to `currentNodeId`, `runnerStatus`, or `accumulator`. This invariant was established in Phase 2 and enforced consistently through Phases 30 and 31.

**When to use:** Any new ProtocolRunner method that responds to user input and changes runner state.

**Source:** 30-CONTEXT.md (D-08, anti-patterns), 31-PATTERNS.md (section 4a), 30-02-SUMMARY.md

### P-02: Append-only service extension â€” new methods, untouched existing ones

`SnippetService.listFolder` was added after the existing `list()` method without modifying `list()` in any way. This preserves backward compatibility (Snippet Manager still uses `list()`) while extending capabilities.

**When to use:** When extending a service with new functionality that serves a different consumer.

**Source:** 30-01-SUMMARY.md (patterns-established), 30-CONTEXT.md (D-21)

### P-03: Pre-I/O path normalization + containment check for attacker-influenced paths

Before any `vault.adapter.exists()` or `vault.adapter.list()` call, validate that the resolved path: (1) contains no `..` or `.` segments, (2) is not absolute, and (3) starts with the allowed root prefix. Reject and return empty results if any check fails.

**When to use:** Any vault operation where the path originates from user-editable canvas JSON or other untrusted input.

**Source:** 30-01-SUMMARY.md (patterns-established), 30-CONTEXT.md (D-20)

### P-04: Two-priority type resolution for field injection

When injecting a derived field (like color), resolve the source value with priority: (1) from the current edits payload (explicit change), (2) from the existing stored record (fallback for field-only saves). This covers both type-change saves and field-only saves in a single flow.

**When to use:** Whenever a save operation must inject a derived field that depends on another field's value, and that other field may or may not be in the current edits.

**Source:** 28-CONTEXT.md (D-04), 28-RESEARCH.md (Architecture Patterns)

### P-05: Partitioned neighbor rendering with typed arrays

For mixed-type branching, partition neighbors into typed arrays (`answerNeighbors`, `snippetNeighbors`) in a single pass, then render each partition as a separate DOM section. This keeps rendering logic clean and avoids interleaved type checks.

**When to use:** When a node can have outgoing edges to multiple node kinds and each kind needs distinct rendering.

**Source:** 31-04-SUMMARY.md, 31-PATTERNS.md (section 6)

### P-06: Empty-string to undefined normalization for optional canvas fields

Canvas JSON optional fields use `v || undefined` in both the parser (`typeof rawPath === 'string' && rawPath !== '' ? rawPath : undefined`) and the editor form (`this.pendingEdits[key] = v || undefined`). This ensures absent-means-default semantics: `saveNodeEdits` deletes keys with `undefined` values.

**When to use:** Any optional field on a canvas node that has fallback/default behavior when absent.

**Source:** 29-02-SUMMARY.md, 29-REVIEW.md (WR-02), 31-01-SUMMARY.md, 31-PATTERNS.md (section 2)

### P-07: Pattern map document for extension-heavy phases

Phase 31 produced a dedicated PATTERNS.md that mapped every file change to its closest existing analog in the codebase, with exact line numbers and code snippets. This made the phase essentially "paint by numbers" â€” every change had a template to copy.

**When to use:** Phases that extend many files with structurally similar changes (adding a node kind, adding a runner status, etc.).

**Source:** 31-PATTERNS.md

## Surprises

### S-01: Removing duplicate color injection from onTypeDropdownChange also fixed the live-path unmark bug

WR-01 (live-save unmark sends `color: undefined`) and WR-02 (duplicate color injection in `onTypeDropdownChange`) turned out to share the same root cause. The `color: undefined` value reaching `saveLive` originated from the duplicate block in `onTypeDropdownChange`. Removing that block (WR-02 fix) eliminated WR-01 as well, requiring no additional code.

**Impact:** Two review warnings resolved with a single deletion. Reinforces that removing duplication is often safer than patching around it.

**Source:** 28-REVIEW-FIX.md (WR-01 resolved by WR-02 fix)

### S-02: Plan's path-normalization sketch was itself a security bug

The Phase 30 plan suggested filtering out `..` segments from paths and then checking containment. But this normalization silently collapses `.radiprotocol/snippets/../../etc` to `.radiprotocol/snippets/etc`, which passes the containment check â€” the exact opposite of what the security gate should do. The TDD test caught this during implementation.

**Impact:** Plans themselves can contain security vulnerabilities. TDD tests serve as an independent verification layer for plan correctness, not just implementation correctness.

**Source:** 30-01-SUMMARY.md (deviation auto-fix)

### S-03: UAT of Phase 30 directly surfaced the Phase 31 feature request

During Phase 30 UAT, the user discovered that snippet nodes could only be chained after answers, not placed as direct branches of a question node. This limitation was immediately captured as a feature request in the UAT gaps section and promoted to Phase 31 (from backlog item 999.3).

**Impact:** Hands-on UAT is an effective feature discovery mechanism, not just a verification step. The most impactful feature of v1.4 (mixed branching) originated from UAT, not from upfront planning.

**Source:** 30-UAT.md (Gaps section), 31-CONTEXT.md (promoted_from: 999.3)

### S-04: The logically dead guard `!isTypeChange && !isUnmarkingType` was hiding an invariant

Phase 28 review found that `!isUnmarkingType` was always true when `!isTypeChange` was true (since `isUnmarkingType` implies `isTypeChange`). The redundant condition was not a bug but signaled a misunderstanding of the relationship between the two flags. Removing it made the invariant explicit in the code.

**Impact:** Dead conditions in guards may indicate unclear domain reasoning. Removing them improves readability and prevents future readers from adding wrong-headed logic.

**Source:** 28-REVIEW.md (WR-03), 28-REVIEW-FIX.md

### S-05: All Phase 31 file changes had exact analogs â€” zero novel patterns needed

Phase 31's pattern map found that every single file change (11/11 files) had an exact or role-match analog already in the codebase. No new patterns, APIs, or architectural decisions were needed. The entire phase was a structured copy-and-adapt exercise.

**Impact:** For extension-heavy phases, investing in a pattern map up front reduces planning and execution time dramatically. Phase 31 had zero deviations from plan.

**Source:** 31-PATTERNS.md (11/11 analogs found)
