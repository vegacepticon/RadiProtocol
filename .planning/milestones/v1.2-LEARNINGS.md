# v1.2 Learnings: Runner UX & Bug Fixes

## Decisions

### D1. Auto-grow textarea via requestAnimationFrame instead of CSS-only

The protocol preview textarea uses JavaScript (`requestAnimationFrame` setting `height = scrollHeight`) combined with `flex: 1 1 auto` CSS, rather than a pure-CSS solution. This approach was chosen because CSS alone cannot reliably auto-size a textarea to its content height across Obsidian themes.

**Rationale:** CSS `auto` height on textareas does not expand to content; JavaScript measurement is required for true content-fit sizing in the Electron webview.

**Source:** Phase 12 VERIFICATION.md (Observable Truth 1), Phase 19 19-01-SUMMARY.md

---

### D2. Click event over pointerdown for canvas node selection

Phase 14 initially used `pointerdown + setTimeout(0)` to detect canvas node clicks, but switched to a direct `click` event handler. The canvas selection Set is committed by Obsidian's own click handler before the plugin's fires, making first-click detection reliable without a timeout hack.

**Rationale:** `pointerdown` fires before Obsidian commits the selection, requiring a `setTimeout(0)` workaround. `click` fires after selection is committed, eliminating the race condition.

**Source:** Phase 14 VERIFICATION.md (Observable Truth 1, Plan 03 fix), Phase 19 19-03-SUMMARY.md

---

### D3. Promise-based modal for unsaved edit guard

The `NodeSwitchGuardModal` returns a `Promise<boolean>` via `modal.result`, allowing `handleNodeClick()` to `await` the user's decision inline. This avoids callback nesting and keeps the guard logic readable.

**Rationale:** Promise-based modals integrate cleanly with async/await control flow, making the guard condition a simple `if (!confirmed) return` one-liner.

**Source:** Phase 14 VERIFICATION.md (Observable Truth 2, Required Artifacts)

---

### D4. Unconditional widget creation for sidebar/tab parity

The `CanvasSelectorWidget` is created unconditionally in `onOpen()` — not gated on sidebar vs. tab mode. This ensures feature parity between display modes without branching logic.

**Rationale:** Gating features on display mode creates maintenance burden and risks feature divergence. Unconditional creation keeps the code simpler and guarantees both modes get the same widget.

**Source:** Phase 13 VERIFICATION.md (Observable Truth 1)

---

### D5. CSS-only phase for gap closure (Phase 18)

Three CSS gaps identified by the milestone audit were consolidated into a single CSS-only phase (Phase 18) with no TypeScript changes. This separated visual styling from functional wiring.

**Rationale:** The TypeScript wiring was already correct from Phases 12-14; only CSS was missing. A dedicated CSS phase avoided unnecessary risk of introducing regressions in working TypeScript code.

**Source:** Phase 18 18-CONTEXT.md

---

### D6. All CSS color values use Obsidian CSS variables

No hard-coded `#hex` or `rgb()` values are used in any CSS rule. All colors reference Obsidian CSS variables (e.g., `var(--interactive-accent)`, `var(--background-modifier-border)`).

**Rationale:** Community review requirement for Obsidian plugins; ensures theme compatibility across light/dark modes and custom themes.

**Source:** Phase 18 18-CONTEXT.md (General constraints)

---

### D7. Run Again button clears session before restart

`restartCanvas()` calls `sessionService.clear(filePath)` before `openCanvas(filePath)`, ensuring no resume modal appears. This was a deliberate choice to provide a clean restart experience.

**Rationale:** Users clicking "Run Again" expect a fresh start, not a resume prompt. Clearing the session data before reopening guarantees this.

**Source:** Phase 13 VERIFICATION.md (Observable Truth 3), Phase 19 19-02-SUMMARY.md

---

### D8. Evidence reconstruction from source code when planning artifacts missing

Phases 12 and 13 had no PLAN or SUMMARY artifacts preserved. Phase 19 reconstructed all verification evidence from current source code line numbers and git log, explicitly noting this methodology.

**Rationale:** Formal verification should not be blocked by missing planning artifacts. Source code and git history are the ground truth; reconstructed evidence is valid when clearly attributed.

**Source:** Phase 19 19-01-SUMMARY.md, 19-02-SUMMARY.md

## Lessons

### L1. CSS gaps silently accumulate when TypeScript and CSS are delivered in separate sessions

Phases 12 and 13 delivered correct TypeScript wiring but left CSS rules missing. The milestone audit discovered three CSS gaps (LAYOUT-03, SIDEBAR-01, RUNNER-01) that were invisible during functional testing because the features "worked" without styling.

**Impact:** Features can appear functional in testing while being visually broken for users. CSS completeness should be verified alongside functional wiring in every phase.

**Source:** Phase 18 18-CONTEXT.md, Phase 13 VERIFICATION.md (Cross-phase CSS note)

---

### L2. Stale worktree merges can silently revert source changes

A merge commit (`a9cf3aa`) from a stale worktree reverted Phase 13, 14, and 18 source changes. The initial Phase 19 verification run scored 1/8 because the code was actually missing. A dedicated restore commit (`52bb34e`) was required to fix this.

**Impact:** Worktree branches that fall behind main can destructively overwrite newer code on merge. Worktree branches should be rebased or updated before merging.

**Source:** Phase 19 VERIFICATION.md (Context section, re_verification frontmatter)

---

### L3. Dead CSS rules produce no visible harm but cause inspector confusion

Dead CSS rules for `.rp-legend`, `.rp-legend-row`, and `.rp-legend-swatch` remain in `styles.css` even though no DOM elements emit these classes. They produce no visible output but appear in browser developer tools, which can mislead during debugging.

**Impact:** Low — no user-facing effect. Consider periodic dead-CSS cleanup passes to reduce inspector noise.

**Source:** Phase 12 VERIFICATION.md (Observable Truth 4, Gaps Summary)

---

### L4. Early-return guard needed to prevent listener accumulation from modal-triggered events

Opening a modal in Obsidian triggers `active-leaf-change` events, which in turn re-called `attachCanvasListener()`. Without an early-return guard comparing container element references, duplicate click listeners accumulated on the canvas.

**Impact:** Without the guard, each modal open/close cycle added another click listener, causing duplicate `handleNodeClick()` calls and potential double-modal issues.

**Source:** Phase 14 VERIFICATION.md (Observable Truth 3, Required Artifacts)

---

### L5. Planning artifact preservation matters for future verification

Phases 12 and 13 had no PLAN or SUMMARY files preserved — only a one-line COMPLETED.md. Phase 19 spent significant effort reconstructing evidence from source code and git log that would have been trivially available from preserved artifacts.

**Impact:** Future phases should always preserve PLAN and SUMMARY artifacts, even for small phases. The cost of preservation is near-zero; the cost of reconstruction is significant.

**Source:** Phase 19 19-01-SUMMARY.md ("Phase 12 planning artifacts were not preserved"), 19-02-SUMMARY.md ("Phase 13 produced only COMPLETED.md")

---

### L6. Selector bar needs re-prepend after contentEl.empty()

The `render()` method calls `contentEl.empty()` which destroys all children, including the selector bar. The selector bar element must be re-prepended after every `contentEl.empty()` call to survive re-renders.

**Impact:** Without the re-prepend at line 275, the canvas selector disappears after any state change that triggers a re-render.

**Source:** Phase 13 VERIFICATION.md (Observable Truth 1, Required Artifacts)

## Patterns

### P1. Cross-phase CSS attribution pattern

When functional TypeScript wiring is delivered in one phase but CSS is added in a later phase, verification documents should explicitly attribute CSS to the phase that added it. This prevents confusion about which phase is responsible for which artifact.

**When to use:** Any time a verification document references code or CSS that was not delivered by the phase being verified.

**Source:** Phase 19 19-02-SUMMARY.md (Decision 2), Phase 12 VERIFICATION.md (Note on LAYOUT-03)

---

### P2. DOM creation order equals visual order in flex-column containers

In a `flex-direction: column` container, visual order matches DOM creation order. This means layout requirements (e.g., "question zone below preview zone") can be verified by checking the order of `createDiv` calls in the render method without needing to inspect CSS `order` properties.

**When to use:** Verifying vertical layout ordering in Obsidian plugin views that use flex-column containers.

**Source:** Phase 12 VERIFICATION.md (Observable Truth 2), Phase 19 VERIFICATION.md (Observable Truth 2)

---

### P3. Grep-based validation for CSS-only phases

For phases that only add CSS rules (no TypeScript), validation can be reduced to grep commands that check for the presence of expected selectors in the stylesheet. This provides fast (<2 second) feedback without requiring a test framework.

**When to use:** CSS-only phases where the concern is whether rules exist, not their visual rendering.

**Source:** Phase 18 18-VALIDATION.md (Test Infrastructure, Sampling Rate)

---

### P4. Same-node no-op guard prevents flicker

When implementing click-to-load behavior, always include a same-node check (`if (this.currentNodeId === nodeId) return`) to prevent re-loading the same content on repeated clicks. This eliminates visual flicker and unnecessary state changes.

**When to use:** Any click-to-load or click-to-select pattern where repeated clicks on the same item are possible.

**Source:** Phase 14 VERIFICATION.md (Observable Truth 1, line 100 same-node no-op guard)

---

### P5. Formal verification reconstruction from source code

When planning artifacts are missing, formal verification can be reconstructed by: (1) grepping current source for expected function/class/selector names, (2) citing exact line numbers as evidence, (3) checking git log for relevant commits, and (4) explicitly noting the reconstruction methodology.

**When to use:** Retroactive verification of phases that did not preserve PLAN/SUMMARY artifacts.

**Source:** Phase 19 19-01-SUMMARY.md (Evidence reconstruction approach decision)

## Surprises

### S1. Obsidian modals trigger active-leaf-change events

Opening a modal dialog in Obsidian fires `active-leaf-change` workspace events. This was unexpected because modals are typically considered overlays, not leaf changes. This caused listener accumulation in Phase 14 until an early-return guard was added.

**Impact:** Any workspace event listener registered in a view must account for modal-triggered spurious events, or risk duplicate registrations.

**Source:** Phase 14 VERIFICATION.md (Observable Truth 3, Plan 03 fix)

---

### S2. Canvas selection Set is committed on click, not on pointerdown

Obsidian's internal canvas implementation commits the node selection Set during the `click` event handler, not during `pointerdown`. This means plugins using `pointerdown` to detect selection will see stale selection state and need workarounds (e.g., `setTimeout(0)`).

**Impact:** Changed the implementation approach from `pointerdown + setTimeout(0)` to direct `click` event, simplifying the code significantly.

**Source:** Phase 14 VERIFICATION.md (Observable Truth 1), Phase 19 19-03-SUMMARY.md

---

### S3. Stale worktree merge reverted three phases of work

A merge from a stale worktree branch (`a9cf3aa`) silently reverted all source changes from Phases 13, 14, and 18. The verification run that discovered this scored 1/8 instead of the expected 8/8, leading to a dedicated restore commit.

**Impact:** High — three phases of work were temporarily lost. This motivated more careful worktree management practices.

**Source:** Phase 19 VERIFICATION.md (Context section, re_verification frontmatter)

---

### S4. CSS-only phase closed three gaps across two earlier phases

Phase 18, a single CSS-only phase, closed gaps originating from two different feature phases (12 and 13). The TypeScript code was fully correct in all cases — only CSS styling was missing. This revealed that the original phases had been considered "complete" despite missing visual implementation.

**Impact:** Reinforced the need to verify CSS completeness as part of phase completion criteria, not just functional correctness.

**Source:** Phase 18 18-CONTEXT.md, Phase 19 VERIFICATION.md (Requirements Coverage table)
