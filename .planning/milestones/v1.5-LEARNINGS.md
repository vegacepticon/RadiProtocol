# v1.5 Learnings: Snippet Editor Refactoring

## Decisions

### D1. Discriminated union `Snippet = JsonSnippet | MdSnippet` matching the existing RPNode pattern

Introduced a `kind: 'json' | 'md'` discriminated union for snippets, forcing every consumer to branch by kind at compile time. The `SnippetFile` type alias was retained as `type SnippetFile = JsonSnippet` to minimize callsite churn during the transition.

**Rationale:** Maximum type safety with minimal disruption. The `RPNode` discriminated union was already established in the codebase, so following the same pattern kept the architecture consistent and let TypeScript catch every unhandled branch.

**Source:** Phase 32 32-CONTEXT.md (D-01), 32-00-SUMMARY.md

---

### D2. Path = identity; id field deprecated

Full vault-relative path (including file extension) became the sole identifier for snippets. The legacy `id` field in JSON files is tolerated on disk but ignored at runtime. Basename is derived from the file path, never from internal JSON fields.

**Rationale:** Eliminates the dual-identity problem where `id` and filename could diverge. During Phase 33 UAT, a bug surfaced where `handleSave` used `draft.id ?? draft.name` to build paths, silently breaking after `listFolder` stopped populating `id`. Path-as-identity prevents this entire class of bugs.

**Source:** Phase 32 32-CONTEXT.md (D-02), Phase 33 33-05-SUMMARY.md (gap fix commit `1d25985`)

---

### D3. Service layer stays I/O-only; view orchestrates canvas-ref-sync

`SnippetService` methods (`moveSnippet`, `moveFolder`, `renameSnippet`, `renameFolder`) perform only vault I/O. The view layer calls `rewriteCanvasRefs` after successful I/O and controls the user-facing Notice.

**Rationale:** Matches the Phase 33 convention (D-09), simplifies service-layer testing (no canvas mock needed), and gives the view full control over UX feedback. Established in Phase 33 and carried through Phase 34.

**Source:** Phase 34 34-CONTEXT.md (D-01), Phase 33 33-03-SUMMARY.md

---

### D4. Canvas-ref mapping uses snippet-root-relative paths without extension

`rewriteCanvasRefs` mapping keys are snippet-root-relative (e.g., `a/b`, not `.radiprotocol/snippets/a/b.json`). The `toCanvasKey` helper normalizes vault-relative paths to this format. File renames/moves do NOT trigger canvas-ref-sync because `SnippetNode` stores only folder paths, not filenames.

**Rationale:** Discovered during Phase 34 planning that Phase 33's D-09 call was a placebo -- it passed vault-relative paths with extensions that could never match the on-disk `radiprotocol_subfolderPath` format. This was the single highest-priority research finding in the milestone and required retroactive cleanup.

**Source:** Phase 34 34-CONTEXT.md (D-03), 34-PATTERNS.md (pre-plan open question), 34-04-SUMMARY.md

---

### D5. `vault.trash(file, false)` for all deletions (Obsidian .trash/)

All snippet and folder deletions use `vault.trash(file, false)` to move files to Obsidian's `.trash/` directory rather than permanent deletion. This works across desktop and mobile, survives vault sync, and is recoverable.

**Rationale:** Matches Obsidian defaults, provides undo safety, and works within Mobile sandbox constraints.

**Source:** Phase 32 32-CONTEXT.md (D-08), 32-01-SUMMARY.md (SC-3)

---

### D6. `vault.rename` over `fileManager.renameFile` for snippet moves

Used the lower-level `app.vault.rename(abstractFile, newPath)` instead of `app.fileManager.renameFile`. Snippets are not referenced by markdown links (they are referenced via `radiprotocol_subfolderPath` in `.canvas` nodes), so `fileManager.renameFile`'s link-update pass would be wasted work.

**Rationale:** Snippets have their own reference system (canvas node metadata). The standard Obsidian rename API for markdown links adds unnecessary overhead and complexity for files that don't participate in the markdown link graph.

**Source:** Phase 34 34-PATTERNS.md (No Analog Found section), 34-CONTEXT.md (D-02)

---

### D7. Unified SnippetEditorModal with mode flag instead of separate create/edit modals

A single `SnippetEditorModal extends Modal` with `mode: 'create' | 'edit'` handles all snippet editing. Differences are limited to title copy, JSON/MD toggle availability (create only), and save pipeline branching.

**Rationale:** Reduces duplication and keeps the editor in one place. Phase 34 later needed to decompose the save pipeline into three branches (save-in-place / move / rename), which would have been duplicated across two separate modals.

**Source:** Phase 33 33-CONTEXT.md (D-05), 33-03-SUMMARY.md

---

### D8. JSON-to-MD toggle is create-only; type is locked on edit

The type toggle is disabled/hidden in edit mode. Converting JSON to MD would need to serialize chip placeholders into text (lossy), and converting MD to JSON would need to parse free text into placeholders (impossible without a schema).

**Rationale:** Preventing lossy/impossible conversions at the UI level. Users who want to convert delete the old snippet and create a new one.

**Source:** Phase 33 33-CONTEXT.md (D-06)

---

### D9. HTML5 DnD with custom MIME types to avoid cross-origin interference

Used `application/x-radi-snippet-file` and `application/x-radi-snippet-folder` custom MIME types. `dragover.preventDefault()` is called ONLY when `dataTransfer.types.includes(...)` contains the custom MIME, so drops from the OS, chip editor, and other sources are not intercepted.

**Rationale:** Without MIME gating, the tree view would intercept every drag operation in Obsidian, including the built-in file explorer and chip editor drag-and-drop.

**Source:** Phase 34 34-CONTEXT.md (D-04), 34-02-SUMMARY.md

---

### D10. F2 inline rename uses `settled` flag to prevent Enter+blur double-commit

The transient `<input>` element uses a closure-scoped `settled` boolean flag. Enter sets `settled = true` and commits; the subsequent blur fires but short-circuits on `settled === true`. Escape also sets `settled = true` before cleanup.

**Rationale:** Without this guard, pressing Enter triggers both the keydown commit and the subsequent blur commit, resulting in a duplicate rename attempt (which throws on collision check). This is a known HTML input lifecycle race condition.

**Source:** Phase 34 34-CONTEXT.md (D-05), 34-03-SUMMARY.md

---

### D11. MD snippet insertion is verbatim with no content transformation

`mdSnippet.content` is inserted byte-for-byte via `completeSnippet(snippet.content)`. No frontmatter stripping, no whitespace normalization, no separators between blocks.

**Rationale:** User explicitly stated "in MD snippets there will be only the needed text, nothing else." Adding transformation logic would be complexity with no user benefit in the current workflow.

**Source:** Phase 35 35-CONTEXT.md (D-02)

---

### D12. Modal save delta decomposition into three branches

`SnippetEditorModal.handleSave` decomposes changes into three independent branches: save-in-place (content only), `moveSnippet` (folder changed), and `renameSnippet` (basename changed). Each branch has its own Notice text.

**Rationale:** Phase 34's initial implementation routed all non-identity saves through `moveSnippet`, which silently dropped pure-name renames (basename change without folder change). Decomposing the delta fixed this and made each operation independently testable.

**Source:** Phase 34 34-05-SUMMARY.md (post-UAT fix `fd0d50d`)

## Lessons

### L1. Breaking API changes cascade further than planned -- always include callsite shims in the same plan

Phase 32 Plan 01 scoped its changes to `snippet-service.ts` only, but the breaking API changes (removed `list()`, changed `load(id)` to `load(path)`) immediately broke three other files. The executor had to apply Rule 3 blocking shims in `runner-view.ts`, `snippet-manager-view.ts`, and tests to maintain a green build.

**Context:** This happened in every plan that touched shared service APIs. The recurring pattern suggests that plans modifying service signatures should always budget for callsite updates in the same plan, not defer them.

**Source:** Phase 32 32-01-SUMMARY.md (Deviations from Plan), 32-00-SUMMARY.md (Auto-fixed Issues)

---

### L2. Tests written by executor agents may compile but violate strict TypeScript options

Phase 32 Plan 04 wrote 58 passing vitest tests, but the tests contained 17 TypeScript strict-mode violations (`noUncheckedIndexedAccess`, `strictNullChecks`) that broke `npm run build`. The executor only ran `npm test` (vitest), not `npm run build` (which includes `tsc -noEmit`).

**Context:** When `tsconfig.json` includes test files in its `include` pattern, test code must satisfy the same strict options as production code. Executor instructions should mandate running `npm run build` (not just `npm test`) as the final gate.

**Source:** Phase 32 32-VERIFICATION.md (Gaps Summary), Phase 33 33-04-SUMMARY.md (Rule 3 blocking fix for pre-existing TS errors from Plan 03)

---

### L3. Mock DOM diverges from real DOM -- `parentElement` vs `.parent`

Phase 34's `startInlineRename` used `(labelEl as any).parent` to find the row element, which worked in mock-DOM tests but failed in real Obsidian. Real DOM uses `parentElement`. The fix was a three-tier lookup: `parentElement` first (real DOM), `.parent` second (mock fallback), `labelEl` last-resort.

**Context:** Any code that traverses DOM parents must be tested with awareness that mock DOM objects may have different property names than real DOM. This gap was caught only during human UAT, not automated tests.

**Source:** Phase 34 34-05-SUMMARY.md (post-UAT fix `77b62c1`), 34-UAT.md (Part D context)

---

### L4. Phase 33 D-09 canvas-ref-sync call was a placebo -- never actually matched

Phase 33's `SnippetEditorModal.handleSave` passed vault-relative paths with extensions to `rewriteCanvasRefs`, but the canvas-ref-sync module expects snippet-root-relative paths without extensions. The Notice "Updated canvases: N" always showed N=0. This was discovered only during Phase 34 research when the patterns file flagged the inconsistency.

**Context:** When two modules use different path conventions, a dedicated normalization helper (`toCanvasKey`) should be introduced immediately, not deferred. Integration tests with real canvas fixtures would have caught this at Phase 33 time.

**Source:** Phase 34 34-CONTEXT.md (D-03,Следствие 1), 34-PATTERNS.md (CRITICAL pre-plan open question)

---

### L5. Vault watcher debounce timer must be explicitly cleared in `onClose`

Even though `registerEvent` auto-detaches vault event listeners on view close, an already-scheduled `setTimeout` from before close will still fire. Phase 33 explicitly clears the pending redraw timer in `onClose` to prevent a rebuild after the view is destroyed.

**Context:** Auto-detach mechanisms (like Obsidian's `registerEvent`) handle subscription cleanup but not already-enqueued callbacks. Any debounced callback pattern needs explicit timer cleanup in the teardown path.

**Source:** Phase 33 33-04-SUMMARY.md (interpretive decisions, point 4), 33-VERIFICATION.md (SYNC-03)

---

### L6. Chip editor extraction requires `skipName` option to avoid duplicate fields

When the chip editor was extracted from the legacy snippet-manager-view and mounted inside the new modal, it rendered its own English "Name" field on top of the modal's Russian "Name" field. The fix was adding a `MountChipEditorOptions.skipName` flag so the modal passes `{ skipName: true }`.

**Context:** When extracting UI components from a monolithic view into reusable modules, the extracted component may render fields that the new host already provides. Always audit for duplicate fields when composing UI from extracted parts.

**Source:** Phase 33 33-05-SUMMARY.md (gap fix #2, commit `1d25985`)

---

### L7. SnippetService.listFolder and load must use basename as authoritative name, not JSON inner field

Externally renaming a `.json` snippet in Obsidian's file explorer left the old name in the tree because `listFolder` fell back to the JSON file's inner `name` field. The D-02 contract comment in the source already stated "basename is authoritative" but the implementation was out of sync.

**Context:** When a design document declares "X is the source of truth," the implementation must be verified against that declaration, not just against the test suite. This gap was caught during human UAT, not automated tests.

**Source:** Phase 33 33-05-SUMMARY.md (gap fix #1, commit `1d25985`), 33-VERIFICATION.md (Truth 5, post-UAT fix)

---

### L8. F2 hotkey may conflict with native Obsidian commands

During Phase 34 UAT, the user discovered that F2 was already bound to a native Obsidian command on their installation. The ПКМ (right-click) menu "Rename" entry works unconditionally on any installation, while F2 depends on no conflicting keybinding.

**Context:** Plugin keyboard shortcuts should always have an alternative entry point (e.g., context menu) because users may have conflicting hotkeys. Document potential conflicts in user-facing help.

**Source:** Phase 34 34-UAT.md (Part D context), 34-05-SUMMARY.md

---

### L9. Source-level grep assertions are brittle but effective as interim contract tests

Phase 35 tests read `runner-view.ts` source via `fs.readFileSync` and assert on regex patterns (e.g., `snippet.kind === 'md'`). This is necessary because `RunnerView` cannot be mounted in the test suite (no jsdom harness). The patterns break on code reformatting but serve as a strong interim contract until a proper view-layer test harness exists.

**Context:** When testing view-layer code without a DOM harness, source-level assertions fill the gap but should be documented as interim and replaced when proper integration tests become feasible.

**Source:** Phase 35 35-01-SUMMARY.md (Design note), 35-REVIEW.md (IN-01)

---

### L10. Expand-state paths must be rewritten on folder rename/move to avoid ghost-expand

When a folder is renamed or moved, its path in `settings.snippetTreeExpandedPaths` becomes stale. Without rewriting these paths, the tree would try to re-expand a non-existent folder path after the next rebuild. Phase 34 walks the array and rewrites every entry matching the old path prefix.

**Context:** Any persisted UI state that references file/folder paths must be updated when those paths change. Folder deletion should also remove the path from the expanded set (Phase 33 learned this separately).

**Source:** Phase 34 34-03-SUMMARY.md (expand-state prefix rewrite), 33-04-SUMMARY.md (folder delete drops expanded path)

## Patterns

### P1. Pre-I/O path-safety gate via `assertInsideRoot` on every public method

Every path-accepting method in `SnippetService` begins with `const normalized = this.assertInsideRoot(path)`. The helper rejects paths containing `..`, absolute paths outside the snippet root, and paths that escape the root via normalization. Applied uniformly across `listFolder`, `load`, `save`, `delete`, `exists`, `createFolder`, `deleteFolder`, `renameSnippet`, `moveSnippet`, `renameFolder`, `moveFolder`.

**When to use:** Any service method that accepts a user-influenced file path and performs I/O. The gate should be the first statement in the method.

**Source:** Phase 32 32-CONTEXT.md (D-10), 32-01-SUMMARY.md, Phase 34 34-00-SUMMARY.md

---

### P2. WriteMutex.runExclusive wrapping every write operation keyed by normalized path

Every file-write, delete, and rename in `SnippetService` is wrapped in `this.mutex.runExclusive(normalizedPath, async () => { ... })`. Canvas-ref-sync uses its own module-level `canvasMutex` for `vault.modify` calls on canvas files.

**When to use:** Any vault write operation that could race with concurrent writes to the same file (e.g., rapid saves, drag-and-drop bursts, vault watcher triggering a rebuild during a write).

**Source:** Phase 32 32-CONTEXT.md (D-11), 32-02-SUMMARY.md, Phase 34 34-PATTERNS.md

---

### P3. Modal Promise-result pattern with `safeResolve` double-guard

All modals (`SnippetEditorModal`, `ConfirmModal`, `FolderPickerModal`) expose a `readonly result: Promise<T>` field, populated by a `safeResolve(value)` helper that checks a `private resolved = false` flag. `onClose()` always calls `safeResolve(defaultValue)` to prevent unresolved promises. External callers use `const m = new Modal(...); m.open(); const r = await m.result;`.

**When to use:** Every Obsidian Modal subclass that returns a user decision to the caller. The double-guard prevents both double-resolve (from Enter+close race) and unresolved promises (from Esc or overlay click).

**Source:** Phase 33 33-PATTERNS.md (Modal Promise-result pattern), 33-02-SUMMARY.md

---

### P4. `registerEvent` for vault subscriptions + `registerDomEvent` for DOM listeners in ItemView

Using `this.registerEvent(this.app.vault.on(...))` auto-detaches vault listeners on view close (SYNC-03). Using `this.registerDomEvent(el, type, handler)` auto-detaches DOM listeners. Never use raw `vault.on` without `registerEvent` -- causes listener leaks.

**When to use:** Every event listener in an ItemView or WorkspaceLeaf-scoped component. Exception: transient elements (like inline rename `<input>`) use raw `addEventListener` + explicit `removeEventListener` because the element is manually destroyed before the view closes, and `registerDomEvent` would leave stale entries in the view's bookkeeping.

**Source:** Phase 33 33-PATTERNS.md (Obsidian-native event registration), 34-03-SUMMARY.md (raw listeners for transient input)

---

### P5. Shared `performMove` orchestrator funneling multiple UI entry points

Phase 34 extracted a single `performMove(srcPath, srcKind, dstFolder)` method that both the context-menu "Move to..." flow and the DnD drop handler call. All service calls, canvas-ref-sync, expand-state rewrite, and Notice building happen in one place.

**When to use:** When multiple UI entry points (context menu, drag-and-drop, modal, keyboard shortcut) need to perform the same multi-step operation. A shared orchestrator prevents logic divergence and ensures consistent error handling.

**Source:** Phase 34 34-02-SUMMARY.md (performMove extraction)

---

### P6. Best-effort batch processing with skipped-reason collection

`rewriteCanvasRefs` processes all `.canvas` files in a single pass. If one canvas fails to read/parse/write, it is added to the `skipped` array with a reason, and processing continues. The function never throws; the caller inspects the result and shows a Notice.

**When to use:** Any operation that touches multiple independent files where partial success is acceptable. The pattern provides observability (the skipped list) without blocking the entire operation on one corrupt file.

**Source:** Phase 32 32-CONTEXT.md (D-06), 32-02-SUMMARY.md

---

### P7. Wave-based TDD: RED stubs in Wave 0, GREEN implementation in later plans

Phase 35 (and Phase 33) used a structured TDD approach: Plan 01 creates failing test stubs (`it.skip` or actual RED tests) covering all requirements, then Plan 02 implements the feature to turn them GREEN. This ensures test coverage is designed before implementation and verifiable at each step.

**When to use:** Phases with clear requirements that can be expressed as test assertions before implementation begins. Particularly effective for view-layer changes where the test stubs document the expected contract.

**Source:** Phase 35 35-01-SUMMARY.md (Wave 0 RED), Phase 33 33-01-SUMMARY.md (Wave 0 stubs)

---

### P8. Vault watcher with prefix filter + trailing debounce for tree refresh

Three vault event subscriptions (`create`, `delete`, `rename`) filtered by `filePath === root || filePath.startsWith(root + '/')` with a 120ms trailing debounce via `window.setTimeout`. Multiple events in the window collapse to one full re-list + re-render. The `/` boundary in the prefix filter prevents matching sibling folders.

**When to use:** Any tree or list view that needs to stay synchronized with vault changes. Full re-list is intentionally simpler than surgical DOM patching for small datasets (dozens, not thousands of entries).

**Source:** Phase 33 33-CONTEXT.md (D-13, D-18), 33-04-SUMMARY.md

## Surprises

### S1. Phase 33 D-09 canvas-ref-sync was a placebo the entire time

The `rewriteCanvasRefs` call in `SnippetEditorModal.handleSave` passed vault-relative paths with `.json`/`.md` extensions, but the canvas-ref-sync module expects snippet-root-relative paths without extensions. The call was syntactically correct but semantically a no-op -- the Notice "Updated canvases: 0" was always accurate but for the wrong reason.

**Impact:** This was only discovered during Phase 34 research when the patterns analyst flagged the inconsistency between the two modules' path conventions. Phase 34 Plan 04 retroactively fixed it. Without the patterns analysis step, this would have persisted silently through the entire milestone.

**Source:** Phase 34 34-CONTEXT.md (D-03), 34-PATTERNS.md (CRITICAL pre-plan open question)

---

### S2. Real DOM and mock DOM have different parent-traversal properties

Mock DOM objects in tests use `.parent` for parent traversal, but real browser DOM uses `parentElement`. Phase 34's inline rename implementation worked in all 10 unit tests but failed completely in real Obsidian. The input was appended inside a hidden label element because the parent lookup returned the label itself instead of the row.

**Impact:** Required a post-UAT hotfix. This class of bug is invisible to unit tests and can only be caught by human verification in the real Obsidian environment.

**Source:** Phase 34 34-05-SUMMARY.md (post-UAT fix `77b62c1`)

---

### S3. `moveSnippet` silently drops pure-name renames when folder is unchanged

Phase 34 Plan 04 routed all non-identity modal saves through `moveSnippet(oldPath, newFolder)`, which preserves the original basename. When the user edited ONLY the snippet name (folder unchanged), `moveSnippet` moved the file to itself and the rename was lost. The fix required decomposing the save delta into three independent branches.

**Impact:** Required a post-UAT hotfix. The bug was architecturally caused by treating "move" and "rename" as the same operation when they have distinct semantics (folder change vs. basename change).

**Source:** Phase 34 34-05-SUMMARY.md (post-UAT fix `fd0d50d`), 34-UAT.md (Part E, E25)

---

### S4. `tsconfig.json` including test files makes test code subject to production strict-mode rules

Phase 32's `tsconfig.json` has `include: ["src/**/*.ts"]` which captures `src/__tests__/*.ts`. Test code with `.find()` results used without non-null assertions, array destructuring of mock `.calls[0]`, and direct `.kind` access on potentially-undefined array elements all violated `noUncheckedIndexedAccess` and `strictNullChecks`. The tests passed in vitest but broke `npm run build`.

**Impact:** Blocked all subsequent phases until the 17 strict-mode violations were fixed. The root cause is architectural -- either tests need their own `tsconfig` or test code must be as strict as production code.

**Source:** Phase 32 32-VERIFICATION.md (Gaps Summary)

---

### S5. User had F2 bound to a native Obsidian command, blocking the inline rename feature

During Phase 34 UAT, F2 did not trigger inline rename because the user's Obsidian installation had F2 already bound to a built-in command. This is not a bug but a deployment environment conflict. The right-click context menu "Rename" entry works unconditionally.

**Impact:** Low -- the feature works after unbinding the conflicting hotkey. But it highlights that any keyboard shortcut in a plugin may conflict with user-configured or Obsidian-default bindings. Always provide an alternative non-keyboard entry point.

**Source:** Phase 34 34-UAT.md (Part D context)

---

### S6. Phase 35 required only 1 file change and 12 minutes to implement

Despite being a full phase with 4 requirements, 7 tests, and complete UAT, Phase 35's implementation was confined to a single file (`runner-view.ts`) with minimal edits: removing a JSON-only filter, adding a `kind === 'md'` branch, and inserting glyph prefixes. All infrastructure (MdSnippet model, `listFolder` returning both types, `completeSnippet` accepting any string, undo/session mechanics) had been built in Phases 32-34.

**Impact:** Validates the architecture: the discriminated union + path-based API + verbatim content insertion design from Phase 32 paid off with near-zero implementation cost for the final consumer. The total Phase 35 duration was 27 minutes (15 min test stubs + 12 min implementation).

**Source:** Phase 35 35-02-SUMMARY.md (12 min duration), 35-CONTEXT.md (current state scout)
