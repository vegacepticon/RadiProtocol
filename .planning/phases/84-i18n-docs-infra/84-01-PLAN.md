# Plan 84-01 — Scaffold i18n service and locale files

## Objective

Create a type-safe i18n infrastructure that can be consumed by all UI modules.

## Scope

Create:
- `src/i18n/i18n-service.ts` — `I18nService` class with `t(key, params?)` and locale loading
- `src/i18n/locales/en.json` — English strings (extract from existing Russian + already-English UI)
- `src/i18n/locales/ru.json` — Russian strings (current user-facing copy)
- `src/i18n/index.ts` — re-export for consumers

Modify:
- `src/settings.ts` — add `locale: 'en' | 'ru'` to settings + DEFAULT_SETTINGS + UI dropdown
- `src/main.ts` — instantiate `I18nService` in `onload()`, expose via `plugin.i18n`
- `src/donate/wallets.ts` — replace hard-coded Russian strings with `t()` calls (proof-of-concept)
- `src/settings.ts` — replace hard-coded Russian strings in donate section with `t()` calls

Do NOT yet touch:
- `src/views/*`, `src/snippets/*`, `src/graph/*`, `src/runner/*` — deferred to 84-02/84-03

## Implementation notes

- Use flat key schema: `namespace.subkey`, e.g. `settings.donateHeading`, `donate.invitation`
- `t()` supports `{param}` interpolation.
- Locale change triggers `saveSettings()`; consumers re-read on next render.
- Default locale: `'en'` for new installs. Existing installs get `'ru'` if we detect no locale key (backward compat handled by defaulting to `'ru'` when absent? No — default is `'en'` per REQUIREMENTS, but to avoid jarring existing Russian users, use `'ru'` as default if setting is missing, until they explicitly switch. Simpler: default `'en'`; existing users see English once and can switch — acceptable for a dev plugin.)

Actually, per REQUIREMENTS.md: "default is English for new installs, Russian preserved for existing installs (backward compatibility)". So:
- DEFAULT_SETTINGS.locale = 'en'
- In `onload()` after `loadSettings()`, if `settings.locale` is undefined/missing, set it to 'ru' and save. This preserves existing Russian users.
