  <name>Task 1: Create InlineRunnerModal class with floating DOM shell</name>
  <files>src/views/inline-runner-modal.ts</files>
  <action>Create a new file `src/views/inline-runner-modal.ts` that exports an `InlineRunnerModal` class. This is NOT an Obsidian `Modal` subclass — it is a plain class that manages a floating `HTMLElement` appended to `document.body`. Design decisions per 54-CONTEXT.md:

1. **Constructor** accepts: `app: App`, `plugin: RadiProtocolPlugin`, `canvasFilePath: string`, `targetNote: TFile`, optional `startNodeId?: string`.
2. **Private fields**: `private containerEl: HTMLElement` (the floating panel), `private contentEl: HTMLElement` (scrollable content area inside the container), `private headerEl: HTMLElement` (title bar with canvas name + close button), `private runner: ProtocolRunner`, `private graph: ProtocolGraph | null`, `private canvasFilePath: string | null`, `private targetNote: TFile`, `private plugin: RadiProtocolPlugin`, `private activeFileUnsub: () => void` (cleanup for workspace.on('active-leaf-change')), `private fileUnsub: () => void` (cleanup for vault.on('delete')), `private isHidden: boolean` (tracks D1 freeze state).
3. **open()** method: appends `containerEl` to `document.body`, parses canvas via `plugin.canvasParser`, validates via `plugin.validator` (with snippetFileProbe), starts the runner with `runner.start(graph, startNodeId)`, calls `render()`, subscribes to `workspace.on('active-leaf-change')` for D1 freeze/resume, subscribes to `vault.on('delete')` for D2 tab-close detection.
4. **close()** method: unsubscribes all event listeners, removes `containerEl` from DOM, nulls references. Per D3 — no rollback, no confirmation.
5. **render()** method: mirrors RunnerView's `switch(state.status)` pattern but renders into `contentEl` instead of `this.contentEl`. Build the same DOM subtrees: `.rp-preview-zone`, `.rp-question-zone`, `.rp-output-toolbar`. For `at-node`: render question text, answer buttons, snippet branch buttons, skip button (if applicable), step-back button. For `awaiting-snippet-pick`: render snippet picker inline (Task 2 will handle the picker integration). For `awaiting-loop-pick`: render loop picker inline. For `complete`: render final text preview + copy/save/insert toolbar. For `error`: render error message. For `idle`: render loading/starting state.
6. **renderPreviewZone(zone, text)**: creates a `<textarea>` or `<pre>` element showing accumulated text. In inline mode, use a readonly `<pre>` or `<div>` (not a textarea — the note IS the buffer per D4).
7. **renderQuestionZone**: same button rendering logic as RunnerView — answer buttons call `runner.chooseAnswer()`, then `appendAnswerToNote()`, then `render()`. Skip button calls `runner.skip()` then `render()`. Step-back calls `runner.stepBack()` then `render()`.
8. **appendAnswerToNote(text: string)**: the critical inline-specific method. Uses `await app.vault.process(targetNote, content => content + separator + text)` to append the answer text to the end of the note. Separator logic: if note is empty, no separator; otherwise use `plugin.settings.textSeparator` (same as RunnerView's defaultSeparator).
9. **handleActiveLeafChange()**: per D1 — if the active leaf's file is NOT `targetNote`, add `is-hidden` class to `containerEl` (CSS `display: none`). If it IS `targetNote`, remove `is-hidden`. State is preserved — no runner reset.
10. **handleTargetNoteDeleted()**: per D2 — if `targetNote` is deleted (no open leaves), call `close()` to end the run.
11. **Close button in header**: icon `x`, calls `close()` directly (D3 — no confirmation).
12. Follow CLAUDE.md conventions: no `innerHTML`, use `createEl`/`createDiv`, `console.debug` for lifecycle, `console.error` for failures.
13. Do NOT import or extend Obsidian `Modal` — this is a plain class managing its own DOM element.
14. The container should use `position: fixed` (D4) — not `position: absolute` — so it stays visible even when the note scrolls.

Key pattern to follow: look at how `CanvasSelectorWidget` manages its own popover DOM (createEl, event listeners, destroy cleanup) and how `ConfirmModal` uses Promise-based result pattern. InlineRunnerModal is simpler — no Promise result needed; it's fire-and-forget with the note as the output sink.</action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| vault.process() → target note | Untrusted protocol content appended to user's note — content is from canvas author, not external source |
| workspace.on('active-leaf-change') | Event subscription could leak if not cleaned up on close |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-01 | Information Disclosure | appendAnswerToNote | mitigate | vault.process() is atomic per-file — no partial writes; only appends to the specific targetNote passed at launch (D2 binding) |
| T-54-02 | Denial of Service | InlineRunnerModal event subscriptions | mitigate | All subscriptions cleaned in close() via stored unsubscribe functions; onOpen/onUnload lifecycle ensures no leak |
| T-54-03 | Spoofing | Canvas validation | mitigate | Reuses existing GraphValidator with snippetFileProbe — same validation as sidebar/tab mode; invalid canvases rejected before modal opens |
</threat_model>
