<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user click → ProtocolRunner.stepBack | Two synchronous Back invocations from a double-click event burst cross this boundary; the runner must absorb the second one. |
| graph node kind → RunnerView render switch | `node.kind` is a discriminated union; an unhandled kind reaching the at-node arm is an internal invariant violation, not external input. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-66-01-01 | Tampering | `ProtocolRunner.stepBack()` undo stack | mitigate | `_stepBackInFlight` guard + `queueMicrotask` reset ensures a tick-burst of stepBack calls only consumes one undo entry; existing undo-before-mutate snapshot is unchanged. |
| T-66-01-02 | Tampering | `UndoEntry.restoreStatus` payload | mitigate | New field is optional and only consumed in `stepBack`; type narrowed to `RunnerState['status']` so an arbitrary string cannot be assigned without a TS error. |
| T-66-01-03 | Denial of Service | RunnerView render path | mitigate | Replacement of the `Processing...` placeholder with `renderError` + early `return` ensures a hostile graph reaching the dead branch surfaces a visible error rather than locking the user in a silent loop. |
| T-66-01-04 | Spoofing | post-stepBack runnerStatus | accept | `restoreStatus` is set by ProtocolRunner internally at known push sites; no external party mutates the undo stack. Session restore (`restoreFrom`) deep-copies entries — no aliasing. |
| T-66-01-05 | Information Disclosure | `renderError` message | accept | The error message includes only the unexpected `node.kind` string. No PII, no path data. |
</threat_model>
