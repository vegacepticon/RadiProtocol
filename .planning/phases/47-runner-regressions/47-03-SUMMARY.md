## Decisions Made

- **Append-only-per-phase + strictly additive across all three revisions.** CLAUDE.md forbids rewriting sections of shared CSS files and requires regenerating `styles.css` via `npm run build`. All three revisions appended a new block below the previous one with a clear `/* Phase 47: ... revision N */` comment header naming the UAT symptom + root cause + override properties. No pre-Phase 47 property was touched. The final cascade has three stacked Phase 47 blocks per file instead of a single tidy block — accepted because the git log stays clean and each UAT revision is individually revertable via `git revert` if a future regression is traced to that specific property set.
- **UAT revision 1 root cause: Obsidian `box-sizing: content-box` default + implicit width overflow.** The first block (typography only) assumed the button layout box was well-behaved on narrow widths. In narrow-sidebar + tab-mode, the default `box-sizing: content-box` made horizontal padding overflow; loop buttons had no `width` / `min-width` so their intrinsic content width forced their flex parent to exceed the sidebar. Fix: `box-sizing: border-box`, `overflow-wrap: anywhere`, explicit `width: 100%` + `min-width: 0` on loop buttons.
- **UAT revision 2 root cause: Obsidian base-button fixed height + vertical center.** DevTools inspection after revision 1 still failed UAT showed Obsidian's CSS sets `button { height: var(--input-height); display: inline-flex; align-items: center }`, which forced a single-row height and vertically centered the flex content — multi-line wrapped labels extended above and below the visible painted rectangle of the button. Fix: `height: auto` (so the button grows with its content), `align-items: flex-start` (so wrapped lines stack from the top inside the box), horizontal padding bumped to `var(--size-4-4)` (~16px — Obsidian's standard button rhythm for wider breathing room at narrow widths), defensive `white-space: normal` + `max-width: 100%`.
- **Loop buttons picked up the wrap trio they lacked entirely.** Pre-Phase 47 `loop-support.css:63-71` had only text-align + padding + border-radius + border:none + cursor:pointer + font-size — no `white-space`, no `word-break`, no `overflow-wrap`. Multi-word unlabeled body-branch edge captions (e.g. «следующая проверка (базовая)») would therefore not wrap at all under the old rule. Phase 47 block 1 added `white-space: normal` + `word-break: break-word`; revision 1 added `overflow-wrap: anywhere`. This was a de-facto Rule-2 auto-add (missing critical functionality — loop choice buttons were never equipped to wrap their labels, so RUNFIX-03's "at every wrap count" criterion could not even be evaluated on them without this addition). Documented here rather than as a separate deviation because the plan's `<action>` already called for `line-height: 1.55` + `min-height: 44px` on loop buttons, and the wrap trio is the inseparable complement — without it, `line-height` governs invisible single-line text.
- **No TypeScript changes.** CLAUDE.md + the plan both forbade it. The RUNFIX-03 bug is purely presentational; the TypeScript runner and view layers already emit the correct button markup and event handlers — the glyphs were painted outside the button rectangle, not missing. Keeping the fix in the CSS layer also means zero test regressions are possible (confirmed by the full vitest suite staying at 428 passed / 1 skipped).
