| Property | Value |

## Component Inventory

### Settings Tab (`RadiProtocolSettingsTab.display()`)

The full settings tab replaces the existing stub with four groups of controls rendered via the Obsidian `Setting` API.

#### Group 1: Runner

**Heading:** "Runner" — rendered via `new Setting(containerEl).setHeading()`

| Control | Type | Label | Description |
|---------|------|-------|-------------|
| Runner view mode | `addDropdown()` | "Runner view mode" | "Choose where the protocol runner opens. Changes take effect the next time you invoke the runner." |

Dropdown options:
- Value `"sidebar"` → display label `"Sidebar panel"` (default)
- Value `"tab"` → display label `"Editor tab"`

**Setting field name:** `runnerViewMode: 'sidebar' | 'tab'`

#### Group 2: Output

**Heading:** "Output" — rendered via `new Setting(containerEl).setHeading()`

| Control | Type | Label | Description |
|---------|------|-------|-------------|
| Output destination | `addDropdown()` | "Output destination" | "Where to send the completed protocol report." |
| Output folder | `addText()` | "Output folder" | "Vault-relative folder for saved reports. Used when destination is 'Save to note' or 'Both'." |

Output destination dropdown options:
- Value `"clipboard"` → display label `"Copy to clipboard"` (default)
- Value `"new-note"` → display label `"Save to note"`
- Value `"both"` → display label `"Both"`

#### Group 3: Protocol Engine

**Heading:** "Protocol engine" — rendered via `new Setting(containerEl).setHeading()`

| Control | Type | Label | Description |
|---------|------|-------|-------------|
| Max loop iterations | `addText()` | "Max loop iterations" | "Maximum times a loop can repeat before stopping. Prevents infinite loops in misconfigured canvases. Default: 50." |

#### Group 4: Storage (collapsed — advanced)

**Heading:** "Storage" — rendered via `new Setting(containerEl).setHeading()`

| Control | Type | Label | Description |
|---------|------|-------|-------------|
| Snippet folder | `addText()` | "Snippet folder" | "Vault-relative path for snippet JSON files. Default: .radiprotocol/snippets" |
| Session folder | `addText()` | "Session folder" | "Vault-relative path for session JSON files. Default: .radiprotocol/sessions" |

**Layout decision (Claude's discretion):** Four groups with `setHeading()` separators. Four groups are warranted because the settings span four distinct concerns (runner behavior, output, engine limits, storage paths). Using `setHeading()` rather than a custom `<h2>` element is mandatory — the Obsidian community plugin review blocks manual HTML heading insertion in settings.

## Interaction Contract

### Runner View Mode Dropdown — Mode Transition Behavior

- Changing the dropdown value calls `this.plugin.settings.runnerViewMode = value` and `await this.plugin.saveSettings()`.
- No modal confirmation required — D-03 specifies mode takes effect on next invocation only, so there is no immediate destructive consequence.
- No `Notice` needed on change — the description text already communicates the deferred effect.

### `activateRunnerView()` Deduplication (D-04)

No UI elements are added to `RunnerView` itself. The deduplication logic is purely a code branch in `main.ts`. The user interaction is: invoke command → runner appears in the correct location. No visual affordance change in the runner view.

### Settings Tab Input Validation

| Field | Validation | Error feedback |
|-------|-----------|----------------|
| Max loop iterations | Must be a positive integer. If blank or non-numeric, reset to `50` on blur. | No modal — silent reset to default on blur. |
| Output folder | Accept any non-empty string — no path validation (vault.adapter.exists is checked at write time). | None at settings entry time. |
| Snippet folder | Accept any non-empty string. | None at settings entry time. |
| Session folder | Accept any non-empty string. | None at settings entry time. |

## CSS Contract

Phase 8 introduces no new CSS classes. The `Setting` API renders all settings controls using Obsidian's built-in `.setting-item`, `.setting-item-name`, `.setting-item-description`, and `.setting-item-control` classes — no custom styles required.

If any minor spacing adjustments are needed (e.g., extra margin before the first heading), use `containerEl.createEl('div', { cls: 'rp-settings-spacer' })` with a minimal inline style override — but only if Obsidian's defaults produce visual problems. This is not anticipated.

**No additions to `src/styles.css` are expected for this phase.**
