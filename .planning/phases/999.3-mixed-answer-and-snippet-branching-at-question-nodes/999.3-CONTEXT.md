# Phase 999.3 — Mixed Answer + Snippet Branching at Question Nodes

## Vision

Сейчас question-узел имеет outgoing рёбра только к answer-узлам. Snippet-узлы можно подвесить только **после** answer (фаза 30), что вынуждает делать одну ветку = одна snippet-нода. Это негибко.

Цель фазы: разрешить question-узлу иметь **исходящие рёбра одновременно к answer- и snippet-узлам**. На развилке runner показывает все варианты в едином списке выбора. Выбор любого варианта (answer или snippet) ведёт по соответствующей ветке.

Идея зафиксирована из UAT фазы 30.

## Decisions

### D-01: UX варианта на развилке — лейбл

- На snippet-ноде добавляется новое поле **`radiprotocol_snippetLabel`** (string, optional), редактируется в Node Editor.
- В списке вариантов на развилке snippet-вариант показывается как кнопка с текстом этого label.
- Fallback, если label пуст: `📁 Snippet`.

### D-02: UX варианта на развилке — порядок

- Список вариантов на развилке: **answers сверху, snippets снизу**, разделены секциями.
- Snippet-варианты визуально отличаются префиксом `📁` и/или CSS-классом.
- Внутри каждой секции — порядок обхода исходящих рёбер (стабильный, как сейчас для answers).

### D-03: Что вставляется в textarea при выборе snippet-варианта

- **Только отрендеренный текст snippet-шаблона.** Текст question НЕ вставляется. Label snippet-ноды НЕ вставляется.
- Если у snippet есть `{{placeholders}}` — открывается стандартный SnippetFillInModal (как в фазе 30), submit вставляет результат рендера.
- Контракт accumulator симметричен обычному snippet-узлу после answer.

### D-04: Per-node разделитель (separator) для snippet-ноды

- На snippet-ноде добавляется новое поле **`radiprotocol_snippetSeparator`** (enum: `space` | `newline`, optional, default = глобальный text separator из фазы 15).
- Редактируется в Node Editor.
- Применяется при склейке отрендеренного snippet-текста с текущим accumulator. **Перекрывает глобальную настройку** для этой конкретной ноды.
- Применимо как к "branching" snippet-нодам (этой фазы), так и к обычным snippet-нодам после answer (фаза 30).

### D-05: Picker flow при выборе snippet-варианта на развилке

- Полный reuse логики фазы 30: клик snippet-варианта → разветвление исчезает → открывается folder-picker с базы `subfolderPath` этой snippet-ноды → drill-down → выбор snippet → (modal если placeholders) → вставка → runner идёт по исходящим рёбрам snippet-ноды.

### D-06: Кол-во и комбинации

- Question может иметь **любое** число answers + любое число snippets в любой комбинации.
- Допустимо: только snippets, без answers (snippet-only question).
- Каждая snippet-нода — независимый вариант со своим subfolderPath / label / separator.

### D-07: Validator

- `graph-validator` обновляется: question с outgoing рёбрами к snippet-узлам — **разрешено молча**, без warning.
- Question без answers (только snippets) — **разрешено молча**.
- Существующее правило "answer обязан вести к question/end/snippet" не меняется.

### D-08: Step-back из открытого snippet picker (когда picker открыт выбором варианта на развилке)

- Step-back закрывает picker и **возвращает к списку вариантов на развилке** (тому же question), не откатывает дальше.
- Пользователь может выбрать другой вариант (answer или другой snippet) без двойного step-back.
- После выбора варианта в развилке (и до открытия picker) поведение step-back стандартное (откат на предыдущий узел).

### D-09: Session resume

- В runner-state сохраняется: `nodeId` текущего question + (если выбран snippet-вариант и picker открыт) `selectedSnippetVariantNodeId`.
- После reload: если был выбран snippet-вариант — picker открывается на той же snippet-ноде с базовым `subfolderPath`. Drill-down путь во picker'е — view-local, может сброситься (как в фазе 30).
- Если на момент сейва был просто открыт список вариантов (без выбора) — resume показывает список вариантов заново.

## Out of Scope

- Изменение runtime поведения answer-узлов (текст answer вставляется как сейчас).
- Поддержка mixed branching у других типов узлов кроме question.
- Multi-snippet выбор (выбрать несколько snippet-вариантов разом) — нет, выбор всегда один.
- Условные snippet-варианты (показывать snippet только при условии) — отдельная фича, не в этой фазе.

## Open Questions for Researcher

- Как сейчас устроен `RunnerState` для awaiting-answer (где возможны выборы на развилке) — какой shape проще расширить snippet-вариантами?
- Где в runner-state живёт undoStack — как вписать "step-back в развилке закрывает picker, не откатывает узел"?
- Какие изменения нужны в `canvas-parser` для парсинга `radiprotocol_snippetLabel` и `radiprotocol_snippetSeparator`.
- Какие изменения в `editor-panel-view` для UI редактирования двух новых полей snippet-ноды.
- Какие тесты в `graph-validator.test.ts` нужно обновить (mixed branching раньше падал валидатором — какие правила).
