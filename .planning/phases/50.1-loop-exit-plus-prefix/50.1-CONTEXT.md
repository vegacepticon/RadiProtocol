# Phase 50.1: Loop Exit `+` Prefix Convention - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the Phase 49 D-07 convention (`isExitEdge = isLabeledEdge` — any non-empty label = exit) with a **`+`-prefix marker**:

- **Exit edge** = an outgoing edge of a `loop` node whose `label.trim().startsWith('+')`. Caption = `label.trim().slice(1).replace(/^\s+/, '')`.
- **Body edge** = any outgoing edge whose label does NOT start with `+` (labeled or unlabeled — both valid). Runner iteration over body branches is unchanged. Body-button caption = `nodeLabel(target)` per Phase 49 D-11/D-12 (preserved).
- **Validator** rejects loop nodes with zero `+`-edges, ≥2 `+`-edges, zero non-`+` outgoing edges, or a `+`-edge with an empty caption post-strip. Five Russian error texts are locked (see D-04..D-08 below).
- **No auto-migration.** Legacy canvases that used Phase 49's `label: "выход"` (without `+`) surface a dedicated hard validation error directing the user to add the `+` prefix manually.

Phase 50.1 touches `src/graph/node-label.ts` (redefine `isExitEdge`, add `stripExitPrefix`), `src/graph/graph-validator.ts` (LOOP-04 rewrite), `src/runner/protocol-runner.ts` (dispatch continues via `isExitEdge` — behavior changes because `isExitEdge` itself changes), `src/views/runner-view.ts` (exit-caption goes through `stripExitPrefix`). Node Editor is **not** touched (D-17 from Phase 49 carried forward). Phase 50 reconciler / EdgeLabelSyncService / Pattern B writes are **out of scope** — their semantics are preserved verbatim because Phase 50.1 only changes the exit discriminator in `node-label.ts`; the reconciler continues to apply displayLabel to every incoming edge of an Answer, including edges originating at loop body branches — that is the whole reason Phase 50.1 exists.

Phase 50.1 ships **zero CSS changes** (same reasoning as Phase 49 — exit/body button classes already exist).

</domain>

<decisions>
## Implementation Decisions

### Exit Count Policy

- **D-01:** Exactly **one** `+`-prefixed outgoing edge per loop node. Validator rejects ≥2. Rationale: minimal deviation from Phase 49 «exactly one exit» mental model; Runner continues to render a single exit button at the bottom of the loop picker; fewer code paths and tests. If multi-exit becomes useful later, relaxation to ≥1 is a strictly additive follow-up phase.

### Legacy Canvas Migration

- **D-02:** **No auto-migration.** No parse-time rewriting, no on-save upgrade, no one-shot migration command. Canvases using Phase 49's `label: "выход"` without `+` surface a hard validation error in `RunnerView`'s Error panel (same UX surface as Phase 49 D-01/D-02/D-03). Matches project precedent: Phase 49 D-15 and REQUIREMENTS.md Out-of-Scope row 3 (legacy canvas migration). Zero risk of silent mutation on `.canvas` files (especially important given Standing Pitfall #1 on Canvas-view).

- **D-03:** **Dedicated legacy-detection error text.** When a loop node has 0 `+`-edges AND ≥1 non-`+` labeled outgoing edge, emit a distinct error that mentions `+`-prefix migration and lists candidate edge IDs. When 0 `+`-edges AND 0 labeled outgoing edges, emit the generic «нет выхода» error. User gets the right hint for their actual scenario.

### Validator Error Copy (Russian — LOCKED verbatim)

All `{label}` placeholders resolve via `GraphValidator.nodeLabel()` (which delegates to shared `sharedNodeLabel` from `src/graph/node-label.ts`). All `{edgeIds}` placeholders are comma-joined lists of offending edge ids (same shape as Phase 49 D-02 at `graph-validator.ts:113`).

- **D-04 (zero `+`-edges, no other labeled edges — clean zero-exit):**
  ```
  Loop-узел "{label}" не имеет выхода. Пометьте ровно одно исходящее ребро префиксом "+" — текст после "+" станет подписью кнопки выхода.
  ```
- **D-05 (zero `+`-edges, ≥1 labeled non-`+` edge — legacy migration hint):**
  ```

  ```
- **D-06 (≥2 `+`-edges):**
  ```
  Loop-узел "{label}" имеет несколько "+"-рёбер: {edgeIds}. Должно быть ровно одно выходное ребро — уберите "+" с остальных.
  ```
- **D-07 (zero non-`+` outgoing edges — no body):**
  ```
  Loop-узел "{label}" не имеет тела — добавьте исходящее ребро без префикса "+".
  ```
- **D-08 (`+`-edge with empty caption post-strip):**
  ```
  Loop-узел "{label}": "+"-ребро {edgeId} не имеет подписи — добавьте текст после "+".
  ```

Error-check ordering inside `GraphValidator.LOOP-04`: **D-04/D-05** (count `+`-edges → branch on presence of non-`+` labeled) → **D-06** (≥2 `+`-edges) → **D-08** (iterate `+`-edges, assert `stripExitPrefix(label).length > 0`) → **D-07** (body check). Each offending loop node contributes one error per failed check (Phase 49 pattern — multiple errors accumulate; RunnerView Error panel lists all).

### Strip Prefix Rule

- **D-09 (`stripExitPrefix` definition):**
  ```typescript
  function stripExitPrefix(label: string): string {
    return label.trim().slice(1).replace(/^\s+/, '');
  }
  ```
  Semantics:
  - Input is a `+`-prefixed label (caller must verify via `isExitEdge`).
  - Leading whitespace in original label is trimmed first (matches Phase 49 D-06 «trimmed verbatim»).
  - Exactly **one** `+` character is removed.
  - Any whitespace immediately following the `+` is also removed (so `"+ выход"` → `"выход"`, `"+\u00a0выход"` (nbsp) → `"выход"`).
  - `"++foo"` → `"+foo"` (user consciously gets a `+`-prefixed caption; rare).
  - `"+"` alone or `"+ "` (whitespace-only after `+`) → `""` → rejected by D-08 at validator time; Runner never receives a label that would strip to empty.

- **D-10 (`isExitEdge` redefinition):** Replace the current Phase 49 alias `isExitEdge = isLabeledEdge` (at `src/graph/node-label.ts:51`) with a real function:
  ```typescript
  export function isExitEdge(edge: RPEdge): boolean {
    return edge.label != null && edge.label.trim().startsWith('+');
  }
  ```
  `isLabeledEdge` **stays unchanged** — Phase 50 `edge-label-reconciler` uses it as the «edge has a label we can sync» discriminator, and that semantics must not change. Phase 50.1 decouples the two: `isLabeledEdge` = «has any non-empty trimmed label»; `isExitEdge` = «label starts with `+` after trim».

### Runner Dispatch & Caption

- **D-11:** `ProtocolRunner.chooseLoopBranch` (`src/runner/protocol-runner.ts:194`) continues to call `isExitEdge(edge)` — no source change needed because the import stays the same and the semantics are now `+`-based. JSDoc / inline comments updated to reference the new convention.

- **D-12:** `RunnerView` loop-picker arm (`src/views/runner-view.ts:482-506`):
  - CSS class selection: `isExitEdge(edge) ? 'rp-loop-exit-btn' : 'rp-loop-body-btn'` (unchanged logic; `isExitEdge` semantics changed).

  - Click-handler ordering (`capturePendingTextareaScroll` → `syncManualEdit` → `chooseLoopBranch` → `autoSaveSession` → `renderAsync`) preserved character-for-character (RUNFIX-02 + Standing Pitfall invariant).

<specifics>
## Specific Ideas

- The exit button caption strips leading whitespace after `+` so authors can write `"+ выход"` for readability without affecting Runner output — matches how Markdown bullets allow `- item` vs `-item`.
- Legacy-hint error text (D-05) explicitly lists candidate edge IDs ({edgeIds}) so the user can jump straight to the relevant edge in Canvas view without guessing which one is «the exit».
- D-04 and D-05 are **two** distinct branches of «zero `+`-edges», not a generic one-size-fits-all — because the user's fix action is different (D-04: add a `+`-prefixed edge; D-05: edit the existing labeled edge to add `+` to its label).
- D-08 («empty caption post-strip») is emitted **per offending `+`-edge** — one error message per bad edge — so users editing a loop with two empty-`+` edges see both.
- Cyrillic labels like `+выход` work naturally because the strip is a JavaScript `.slice(1)` on a UTF-16 string; `+` is a single BMP code point, so Cyrillic chars after it are never split.
- `unified-loop-labeled-body.canvas` (D-14) is the regression fixture for the exact scenario that motivated Phase 50.1: an Answer node pointed to by both a Question (Phase 50 sync labels the edge) and a loop body branch (Phase 50 also labels that edge). Under Phase 49 this broke validation (the body edge was misclassified as a second exit); under Phase 50.1 validation accepts because the body edge has no `+` prefix.

</specifics>
