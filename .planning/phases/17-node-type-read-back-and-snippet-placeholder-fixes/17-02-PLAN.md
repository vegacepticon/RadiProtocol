<objective>
Fix BUG-04: The "Add" button in the snippet placeholder mini-form has no effect when
clicked — the handler does not fire, the placeholder is not added, and the form stays open.

Root cause (per D-10, UI-SPEC accessibility note): `miniAddBtn` is created without an
explicit `type="button"` attribute. In certain browser/webview contexts, a `<button>`
without an explicit type defaults to `type="submit"`. Even though `addPlaceholderForm` is
a `div` (not a `<form>` element), some webview environments (including Electron's Chromium
used by Obsidian) may exhibit unexpected event behavior with untyped buttons nested inside
interactive containers. Additionally, `miniCancelBtn` has the same omission.

Fix: Add `type="button"` explicitly to both `miniAddBtn` and `miniCancelBtn` using
`setAttribute('type', 'button')` immediately after element creation. This is the
established accessibility pattern per UI-SPEC section "Accessibility".

Verify the full interaction contract described in UI-SPEC section "BUG-04: Snippet
Manager — placeholder mini-form Add button" works end-to-end.

Purpose: Restore correct Add/Cancel behavior in the snippet creator.
Output: Updated snippet-manager-view.ts with typed buttons; UAT checkpoint confirms fix.
</objective>
