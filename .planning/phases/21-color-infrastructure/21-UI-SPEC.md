## No New CSS — Phase Scope Statement

Phase 21 introduces **zero new CSS classes** and **zero new UI components**. The visual change users see (canvas node color changes) is produced entirely by the Obsidian canvas host rendering the `color` field on its node data model. The plugin writes a palette string to that field via `CanvasLiveEditor.saveLive()`. Obsidian handles all rendering.

The existing `.rp-legend-swatch` class in `styles.css` already exists and is **not updated** in this phase. Legend color alignment with the palette is explicitly deferred to a future phase.

## Semantic Color Contract (node-color-map.ts)

This is the normative color contract for the phase. The mapping below defines what `src/canvas/node-color-map.ts` MUST export.

Obsidian canvas palette strings correspond to the following built-in canvas node colors:

| Palette String | Obsidian Canvas Color | Semantic Meaning |
|---------------|-----------------------|-----------------|
| `"1"` | Red | Danger / Loop boundary |
| `"2"` | Orange | Action / Answer |
| `"3"` | Yellow | Content / Text block |
| `"4"` | Green | Start / Go |
| `"5"` | Cyan | Question / Information |
| `"6"` | Purple | Snippet / Code |

### Type-to-Palette Mapping (D-01 from CONTEXT.md)

| Node Type | Palette String | Canvas Color | Rationale |
|-----------|---------------|--------------|-----------|
| `start` | `"4"` | Green | Entry point — "go" semantics |
| `question` | `"5"` | Cyan | Information gathering |
| `answer` | `"2"` | Orange | Action / selection |
| `text-block` | `"3"` | Yellow | Passive content |
| `snippet` | `"6"` | Purple | Code / file insertion |
| `loop-start` | `"1"` | Red | Loop boundary |
| `loop-end` | `"1"` | Red | Loop boundary (intentional share) |

**Intentional exception — loop pair shares red:** `loop-start` and `loop-end` both map to `"1"` (red). This is a deliberate semantic decision: the loop pair is visually treated as a single construct bounded by the same color. This **overrides** ROADMAP success criterion #2 ("no two types share the same colour") by explicit user decision captured in CONTEXT.md D-01 ("По смыслу" / semantic grouping). The planner and executor MUST NOT treat this as a bug.

### Unassigned / Cleared Type

When a node's type is cleared (set to `''` or `undefined`), the canvas node's `color` field MUST be deleted (set to `undefined`). Obsidian will render the node in its default unstyled canvas color. No palette string is written.

## Interaction Contract

This section defines the two interaction paths that constitute the full behavior contract for this phase.

### Path A — Assign Type

**Trigger:** User selects a node type from the Node Editor type dropdown and saves (current save-button path, Phase 23 will replace with auto-save).

**Expected behavior:**
1. `editor-panel-view.ts` save method reads the selected type from the form.
2. Looks up `NODE_COLOR_MAP[selectedType]` → palette string (e.g., `"4"`).
3. Appends `{ color: paletteString }` to the `edits` object passed to `CanvasLiveEditor.saveLive()`.
4. `saveLive()` writes the color field to the in-memory canvas data.
5. **Canvas node color changes immediately** — no canvas close/reopen required.
6. Color is visible to the user within the same interaction frame as the save.

**Fallback (canvas closed / live API unavailable, D-05):** Color write is silently skipped. The `radiprotocol_*` fields still save normally via Strategy A. Color will appear on next canvas open. No error is shown to the user.

**Latency contract:** Color change must be perceptible as immediate (< 200ms from save action to color visible in canvas, assuming canvas is open and live API available).

### Path B — Clear Type (Unmark)

**Trigger:** User clears the node type field in the Node Editor (sets to empty/none) and saves.

**Expected behavior:**
1. `editor-panel-view.ts` save method detects type is empty.
2. Passes `{ radiprotocol_nodeType: '', color: undefined }` to `saveLive()` (or equivalent unmark signal).
3. `saveLive()` unmark path deletes the `color` key from the canvas node data (in addition to removing all `radiprotocol_*` fields — existing behavior).
4. **Canvas node returns to default Obsidian canvas node color immediately** — no canvas reload required.

**Latency contract:** Color removal must be perceptible as immediate (same frame as save, < 200ms).

## Color

Plugin UI colors are all delegated to Obsidian CSS variables. The 60/30/10 split maps to:

| Role | Obsidian Variable | Usage |
|------|-------------------|-------|
| Dominant (60%) | `--background-primary` | Panel backgrounds, textarea |
| Secondary (30%) | `--background-secondary` | Question zone, list panels, sidebar |
| Accent (10%) | `--interactive-accent` | Primary action buttons (Run Again, Loop Again) only |
| Destructive | `--text-warning` / Obsidian Notice | Error badges, orphan indicators |

**Canvas node palette colors** (the core contract of this phase) are not plugin CSS — they are Obsidian canvas built-ins addressed by palette strings `"1"`–`"6"`. See Semantic Color Contract section above.

Accent reserved for: primary CTA buttons only (`rp-run-again-btn`, `rp-loop-again-btn`). Never used for informational indicators or node color coding.
