<objective>
Fix BUG-01: manual text edits typed into the runner preview textarea must survive when the user advances to the next protocol step.

Purpose: Currently, when a user types directly into the runner preview textarea and then clicks an answer button (or submit/loop action), the textarea is re-rendered from runner state — discarding the typed edit. The fix is a capture-before-advance pattern: read the textarea value and inject it into the runner's accumulator before firing each advance action, so the undo snapshot captured inside the action includes the manual edit.

Output:
- `TextAccumulator.overwrite(text)` — replaces buffer without undo semantics (mirrors `restoreTo` semantics, same O(1) string assignment)
- `ProtocolRunner.syncManualEdit(text)` — calls `this.accumulator.overwrite(text)`, pure, no Obsidian imports
- `runner-view.ts` — 3 advance handlers call `this.runner.syncManualEdit(this.previewTextarea?.value ?? '')` before their runner action
- `runner-view.ts` — complete-state toolbar buttons read `this.previewTextarea?.value ?? capturedText` instead of stale `capturedText` closure (per D-03)
</objective>

## Trust Boundaries

|----------|-------------|

| textarea.value → accumulator | Manual edit text flows through syncManualEdit into the protocol text buffer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering | TextAccumulator.overwrite() | accept | Local Obsidian plugin — no network surface; user is trusted; text goes only to clipboard/vault note on explicit user action |
| T-16-02 | Information Disclosure | previewTextarea.value in clipboard/save/insert | accept | User explicitly initiates Copy/Save/Insert; text is the user's own protocol output; no PII from the plugin itself |
| T-16-03 | Denial of Service | Extremely large textarea value (e.g. pasted megabytes) | accept | Textarea is a standard browser element with no enforced limit; Obsidian's vault.modify handles large strings; no infinite loop or crash path introduced by this change |
