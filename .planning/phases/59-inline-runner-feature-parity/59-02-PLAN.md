Wave 1b — Ship INLINE-FIX-04 (configured separator applied when inserting snippet content in inline mode).

Purpose: Close the separator-on-insert parity gap. Today, inline mode calls `appendAnswerToNote(snippet.content)` — passing RAW text bypasses the separator that `runner.completeSnippet()` just prepended to the accumulator. Result: no separator between prior note content and inserted snippet ("glued concatenation" that the user reported in v1.8 UAT).

Root cause is pinpoint-able (59-RESEARCH.md § Summary, HIGH confidence): `inline-runner-modal.ts:812` (MD arm of `handleSnippetPickerSelection`) and `:821` (JSON zero-placeholder arm) both call `appendAnswerToNote(snippet.content | snippet.template)` with the raw snippet text. The correct sink is the accumulator delta — already implemented for the answer-click path at lines 555-576 (`handleAnswerClick`). Mirror that pattern.

Separation of concerns: `ProtocolRunner` owns the separator resolution chain (per-node `radiprotocol_snippetSeparator` → global `defaultSeparator` → `'newline'`). `TextAccumulator.appendWithSeparator` applies the separator. Inline's ONLY job is to read the delta and flush it to the note. This plan does NOT re-implement separator logic in the inline modal; doing so would duplicate the runner's source-of-truth (Pitfall 1: double-separator regression).

private async appendAnswerToNote(text: string): Promise<void> {
  await this.plugin['insertMutex'].runExclusive(this.targetNote.path, async () => {
    const currentContent = await this.app.vault.read(this.targetNote);
    let toAppend = text;
    const sep = this.resolveSeparator();
    if (currentContent.endsWith(sep) && text.startsWith(sep)) {
      toAppend = text.slice(sep.length);
    }
    const newContent = currentContent + toAppend;
    await this.app.vault.modify(this.targetNote, newContent);
  });
}
