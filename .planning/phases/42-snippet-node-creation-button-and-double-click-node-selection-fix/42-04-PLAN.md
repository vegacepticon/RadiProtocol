<objective>
Close UAT gap 2: "Editor toolbar must remain usable at narrow sidebar widths — all 4 buttons (Create question / Create answer / Create snippet / Duplicate) should stay reachable (via wrap, horizontal scroll, or icon-only mode)."

Root cause: `.rp-editor-create-toolbar` in `src/styles/editor-panel.css` (Phase 39, lines 48-57) is a `display: flex; flex-direction: row` container with no `flex-wrap` rule. The default `flex-wrap: nowrap` forces all four buttons onto one row. When the Obsidian right sidebar is narrowed (common for users who prefer a slim panel), the row overflows the panel width and the right-most "Duplicate node" button clips out of the visible area. Phase 42 Plan 02 added a 4th button ("Create snippet node"), which moved the overflow threshold closer to the default sidebar width and made the regression user-visible (UAT Test 4: "при сужении сайдбара крайняя правая кнопка 'Duplicate' уходит за экран").

Fix: append a single new CSS rule at the END of `src/styles/editor-panel.css` that sets `flex-wrap: wrap` on `.rp-editor-create-toolbar`, plus an explicit `row-gap` so the wrapped buttons keep a visible vertical gap (the existing `gap: var(--size-4-1)` is a shorthand for both row-gap and column-gap, but modern browsers honor it; the explicit `row-gap` is set as a belt-and-braces guard against Obsidian themes that override the shorthand).

Rationale for choosing `flex-wrap: wrap` over the alternatives in the UAT "missing" list:
- `overflow-x: auto` requires the user to scroll horizontally to reach the hidden button — extra interaction, not discoverable by users unfamiliar with horizontal scroll in sidebars.
- "Icon-only mode below a breakpoint" requires either a JS media-query observer or a `@container` query and new CSS for icon-only variants; significantly more work than a minor UAT fix warrants.
- `flex-wrap: wrap` is zero-JS, zero-breakpoint, works across all Obsidian viewport widths, and matches the common Obsidian sidebar idiom (several first-party and community panels wrap their control rows).

Scope guarantee: the CSS is append-only per `CLAUDE.md`. The existing `.rp-editor-create-toolbar` rule at lines 48-57 is NOT modified — the new rule SHADOWS the same selector with only the new `flex-wrap` and `row-gap` properties. Per CSS cascade, both rules apply; later declarations win for overlapping properties (but there are no overlaps — the existing rule does not declare `flex-wrap` or `row-gap`).

Purpose: closes UAT gap 2 caused by the 4-button regression introduced in Phase 42 Plan 02.

Output: ~6 lines appended to `src/styles/editor-panel.css` under a `/* Phase 42 Plan 04 */` banner + `npm run build` to regenerate `styles.css`.

Why two properties (and not just `flex-wrap`):
- `flex-wrap: wrap` is the actual functional fix — allows children to break to the next row when they don't fit.
- `row-gap: var(--size-4-1)` mirrors the existing `gap: var(--size-4-1)` from Phase 39. The Phase 39 `gap` shorthand already sets both row-gap and column-gap in modern browsers; the explicit `row-gap` here is defensive — some older Obsidian-shipped Electron versions and some community themes override the `gap` shorthand, and in those environments wrapped rows can collapse to `row-gap: 0`. The explicit rule guarantees the vertical gap is always `var(--size-4-1)` (8px) regardless of cascade interference.

The new rule re-selects `.rp-editor-create-toolbar` with only these two properties. Per CSS cascade specificity-equal-source-order rules, the earlier Phase 39 declarations for `display`, `flex-direction`, `align-items`, `gap`, `padding`, `margin-bottom`, `border-bottom`, `flex` are preserved unchanged; only `flex-wrap` (which the Phase 39 rule does NOT declare — defaults to `nowrap`) is overridden to `wrap`, and `row-gap` (also not declared in Phase 39) is set.
