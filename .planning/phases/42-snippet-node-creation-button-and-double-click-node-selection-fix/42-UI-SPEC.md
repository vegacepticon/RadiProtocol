## Inheritance Statement

Phase 42 is a direct visual extension of **Phase 39 (Quick-Create UI in Node Editor)**. All spacing, typography, color, iconography conventions, and button styling patterns declared in `.planning/phases/39-quick-create-ui-in-node-editor/39-UI-SPEC.md` are inherited **by reference** and MUST NOT be re-derived or re-declared with different values.

This phase adds on top of the existing toolbar:

1. A new primary action button â€” "Create snippet node" â€” visually identical to the existing quick-create buttons, using the accent color family.
2. A new non-interactive helper hint below the node type dropdown when a selected node has no `radiprotocol_nodeType` assigned yet.
3. A behavioral bug fix (double-click node selection) that has no direct visual footprint, but whose post-fix state reuses the helper hint from item 2.

Phase 39's UI-SPEC governs all base token decisions; deviations in this document are flagged explicitly.

### 3. Empty-Type Node Helper Hint (new)

**Location:** Inside `.rp-editor-form`, rendered **directly below the Node type dropdown Setting row** and **above** the `kindFormSection` container. Only appears when the currently loaded node has no `radiprotocol_nodeType` set.

**Class:** `.rp-editor-type-hint`

**Copy (locked, English):** `Select a node type to configure this node`

**DOM contract:**

```html
<p class="rp-editor-type-hint">Select a node type to configure this node</p>
```

**Visual style:**

| Property | Value |
|----------|-------|
| `color` | `var(--text-muted)` |
| `font-size` | `var(--font-ui-smaller)` (~12px) |
| `font-style` | `italic` |
| `margin` | `var(--size-4-1) 0 0 0` (8px top, 0 elsewhere) |
| `padding` | `0` |
| `line-height` | `1.4` |

**Render condition:**

- `currentKind === null` (no `radiprotocol_nodeType` on the node record) â†’ render the hint.
- `currentKind !== null` (any of `start`, `question`, `answer`, `free-text-input`, `text-block`, `loop-start`, `loop-end`, `snippet`) â†’ do NOT render the hint.

**Disappearance trigger:** When the user picks a node type from the dropdown, `renderForm` re-runs (via the existing dropdown `onChange` rebuild path) and the hint is omitted because `currentKind` is now set. The hint does NOT animate â€” it simply does not appear in the re-rendered form.

**Non-goals:** The hint is non-interactive. It is NOT a button, has no click handler, and MUST NOT be styled as a CTA. It is a muted instructional line only.

### 4. Double-Click Bug Fix (behavioral â€” NO new visual surface)

> **Note for the checker:** The double-click fix is a **data-flow / state bug**, not a visual change. No new UI element is introduced for the fix itself. The user-visible *outcome* after the fix is that a freshly double-click-created empty node, when selected, renders the already-specified `renderForm` output â€” which includes the new helper hint from component #3. This is the only visual contract the fix needs.

**Pre-fix symptom (visible error):** Selecting a double-click-created node shows "Node not found in canvas â€” it may have been deleted." inside `.rp-editor-idle`.

**Post-fix expected state:** Selecting a double-click-created empty node opens the editor panel with:
1. The quick-create toolbar (unchanged).
2. The Node type dropdown, value `"â€” unset â€”"` (existing option from Phase 4).
3. The new helper hint: "Select a node type to configure this node".
4. No kind-specific fields (because no type is set).
5. The `.rp-editor-saved-indicator` row, hidden.

No new CSS classes, no animations, no toast, no modal. The success state IS the normal empty-type form.

## Interaction Contract

### Create Snippet Node

| Step | Action | Result |
|------|--------|--------|
| 1 | User clicks `.rp-create-snippet-btn` | `onQuickCreate('snippet')` is invoked |
| 2 | Flow reuses existing Phase 39 path | Flush pending auto-save â†’ `canvasNodeFactory.createNode(canvasPath, 'snippet', currentNodeId)` â†’ 150ms requestSave flush â†’ `loadNode(canvasPath, newNodeId)` (or in-memory `renderForm` bypass, matching Phase 39 Plan 02) |
| 2a | Canvas open, API available | New snippet node appears on canvas adjacent to anchor. Editor panel loads the new node in the snippet-kind form (inherited from Phase 29). |
| 2b | No canvas open | Notice: "Open a canvas first to create nodes." (existing copy from Phase 38 CanvasNodeFactory) |
| 2c | API unavailable | Notice: "Canvas API unavailable â€” update Obsidian to use node creation." (existing copy from Phase 38) |

### Empty-Type Node Selection (post-fix outcome)

| Step | Action | Result |
|------|--------|--------|
| 1 | User double-clicks empty canvas area | Obsidian creates a blank text node (native behavior) |
| 2 | User single-clicks the new node | `handleNodeClick(filePath, nodeId)` fires |
| 3 | `loadNode` â†’ `renderNodeForm` | Reads canvas JSON, finds the node (the bug fix ensures the read sees the new node), detects `radiprotocol_nodeType === undefined` |
| 4 | `renderForm` renders dropdown + hint | Dropdown at "â€” unset â€”", helper hint visible, no kind-specific fields, saved indicator hidden |
| 5 | User picks a type | Dropdown `onChange` â†’ `onTypeDropdownChange` â†’ form re-renders WITHOUT the helper hint |

### Anchor Node Resolution (unchanged from Phase 39)

- If `currentNodeId` is set, pass it as the anchor to `CanvasNodeFactory.createNode`.
- If not set (idle state), pass no anchor â€” factory places the node at canvas origin.

## Implementation Surface

**File:** `src/views/editor-panel-view.ts`

### Toolbar extension

Inside `renderToolbar(container)`, insert a third button between the existing answer button and the duplicate button:

```ts
const sBtn = toolbar.createEl('button', { cls: 'rp-create-snippet-btn' });
sBtn.setAttribute('aria-label', 'Create snippet node');
sBtn.setAttribute('title', 'Create snippet node');
const sIcon = sBtn.createSpan();
setIcon(sIcon, 'file-text');
sBtn.appendText('Create snippet node');
this.registerDomEvent(sBtn, 'click', () => { void this.onQuickCreate('snippet'); });
```

The `onQuickCreate` method already accepts an abstract `kind`. The planner must widen its signature from `'question' | 'answer'` to `'question' | 'answer' | 'snippet'` so the new button can reuse the exact same flush â†’ create â†’ delay â†’ load pipeline that Phase 39 Plan 02 established. No new code path is needed for the snippet branch.

### Helper hint rendering

Inside `renderForm(nodeRecord, currentKind)`, after the Node type dropdown Setting is added and before `this.kindFormSection = formArea.createDiv();`, emit the hint conditionally:

```ts
if (currentKind === null) {
  formArea.createEl('p', {
    cls: 'rp-editor-type-hint',
    text: 'Select a node type to configure this node',
  });
}
```

The existing dropdown `onChange` path already re-renders via `buildKindForm`; however, the helper hint sits OUTSIDE `kindFormSection`, so it will NOT automatically disappear on type change. The planner must either:

- **Option A (preferred):** move the hint emission inside `renderForm` and force a full `renderForm` re-run on type selection (adds one line to the dropdown's `onChange` to call `this.renderForm(nodeRecord, value ? value as RPNodeKind : null)` after `onTypeDropdownChange`). This is the cleanest path and matches the "no new behaviors, reuse existing renders" principle.
- **Option B:** hold a reference to the hint element and call `.remove()` from the dropdown `onChange` when a non-empty value is chosen. Acceptable fallback if Option A breaks debounce semantics.

Either option satisfies the contract as long as the hint disappears the moment the user picks a type.

### Double-click fix

The fix is out of scope for visual UI-SPEC. The planner/executor must diagnose the "Node not found in canvas" error â€” likely causes include stale canvas-JSON read (Obsidian hasn't flushed the double-click-created node to disk yet), in which case the same in-memory bypass used in Phase 39 Plan 02 (`canvas.nodes.get(nodeId).getData()`) can be adopted for `handleNodeClick` â†’ `renderNodeForm` when the disk read misses the node but the in-memory canvas has it. The final state MUST match component #3 above.
