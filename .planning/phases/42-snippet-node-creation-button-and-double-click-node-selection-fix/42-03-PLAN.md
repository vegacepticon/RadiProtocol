<objective>
Close UAT gap 1: "After double-click creates a new canvas node, the Node Editor panel should automatically switch to that newly-created (selected) node so the user can immediately pick its type without extra clicks."

Root cause: `attachCanvasListener` (src/views/editor-panel-view.ts lines 54-101) registers a single `click` event on the canvas container. When the user double-clicks an empty canvas area, Obsidian creates a new text node AND sets it as the sole selection — but Obsidian updates `canvas.selection` **after** the pointer event, not synchronously. The documented contract for this is inside `src/types/canvas-internal.d.ts` at the `selection` field comment: *"Read only AFTER a setTimeout(0) deferral inside a pointerdown handler; Obsidian updates this Set after the pointer event, not synchronously."* The current handler reads `selection` synchronously on `click`, so when the click event ends, the new node is not yet in the selection Set — `size !== 1` short-circuits, and `handleNodeClick` is never invoked. The user then has to click off the node and click it again to trigger a fresh (now-synchronous) selection update — two extra clicks.

Fix pattern (two surgical changes, both in `attachCanvasListener`):

1. Defer the `canvas.selection` read via `setTimeout(() => { ... }, 0)` inside the handler body. This matches the type declaration's own documented contract. For the single-click case (already working), the deferral is harmless — the selection is already set, and one tick of setTimeout is imperceptible. For the double-click-creates-node case (broken), the deferral lets Obsidian's selection update complete before we read.

2. Also register the same handler on `dblclick` (in addition to the existing `click`). This covers the edge case where Obsidian fires `dblclick` WITHOUT a preceding `click` event for the new node (native DOM semantics vary — `dblclick` implies two `click` events in a row, but Obsidian canvas may swallow one).

Both changes use `setTimeout(0)` (NOT `queueMicrotask` — the type doc specifies setTimeout(0), presumably because Obsidian's internal selection update also goes through a macrotask queue). No new factory calls, no new event types, no new pipeline code. Once the handler fires with a valid single-node selection, the existing `handleNodeClick` -> `loadNode` -> `renderNodeForm` -> Phase 42 Plan 01 in-memory fallback pipeline takes over and auto-loads the new node's form (including the `— unset —` dropdown + the empty-type helper hint).

  const canvasLeaves = this.plugin.app.workspace.getLeavesOfType('canvas');
  const activeLeaf = this.plugin.app.workspace.getMostRecentLeaf();

  if (!canvasLeaf) return;

  const canvasLeafInternal = canvasLeaf as unknown as {
    containerEl: HTMLElement;
    view: unknown;
  };

  if (this.watchedCanvasContainer === canvasLeafInternal.containerEl) return;

  this.canvasPointerdownHandler = null;
  this.watchedCanvasContainer = null;

  const canvasView = canvasLeaf.view as unknown as {
    file?: { path: string };
    canvas?: { selection?: Set<{ id: string; [key: string]: unknown }> };
  };

  this.watchedCanvasContainer = canvasLeafInternal.containerEl;

    // BUG: selection is read SYNCHRONOUSLY — violates the type contract above.

    if (!filePath) return;
    void this.handleNodeClick(filePath, node.id);
