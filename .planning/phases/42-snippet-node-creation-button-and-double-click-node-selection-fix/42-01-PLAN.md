<objective>
Fix the "Node not found in canvas" error that appears when a user double-clicks the canvas to create an empty text node and then single-clicks that new node. Add the "Select a node type to configure this node" helper hint specified by the UI-SPEC for empty-type nodes.

Root cause: `renderNodeForm` (src/views/editor-panel-view.ts lines 259-295) reads the canvas file from disk via `vault.read()` and calls `canvasData.nodes.find(...)`. When Obsidian's double-click-created node has not yet been flushed to disk (Obsidian's save cycle is debounced), the disk JSON does not contain the new node, and the `nodeRecord` lookup fails -> `renderError('Node not found in canvas - it may have been deleted.')`.

Fix pattern (established by Phase 39 Plan 02 for `onQuickCreate`): when the disk read misses the node, fall back to the live canvas in-memory state via `canvas.nodes.get(nodeId).getData()`. The `CanvasInternal` type in `src/types/canvas-internal.d.ts` already declares `nodes: Map<string, CanvasNodeInternal>` and `getData(): Record<string, unknown>`, so no new type surface is needed.

Helper hint: after the dropdown renders inside `renderForm` (line 303-332), conditionally emit a `<p class="rp-editor-type-hint">` immediately before `this.kindFormSection = formArea.createDiv();` when `currentKind === null`. The dropdown `onChange` must be extended to call `this.renderForm(nodeRecord, value ? value as RPNodeKind : null)` AFTER the existing `onTypeDropdownChange(value)` call so the hint disappears when a type is picked (UI-SPEC Option A - preferred).

Purpose: closes v1.6 phase 42 success criteria #2 and #3. Reuses the in-memory canvas read pattern already proven in Phase 39 / 40.
Output: patched `renderNodeForm` with fallback, patched `renderForm` with hint + re-render, new CSS rule, 2 new unit tests.
</objective>
