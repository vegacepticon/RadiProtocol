## Directory Purposes

**`src/graph/`:**
- Purpose: Pure graph parsing and validation -- the "model" for protocol definitions
- Contains: Type definitions, canvas JSON parser, structural validator
- Key files: `graph-model.ts` (all RPNode types), `canvas-parser.ts` (the parser)
- Constraint: Zero Obsidian API imports -- must remain pure for unit testing

**`src/runner/`:**
- Purpose: Pure protocol execution state machine -- the "engine"
- Contains: State machine, state types, text accumulator
- Key files: `protocol-runner.ts` (all traversal logic), `runner-state.ts` (discriminated union)
- Constraint: Zero Obsidian API imports -- must remain pure for unit testing

**`src/canvas/`:**
- Purpose: Bridge between plugin logic and Obsidian's internal canvas APIs
- Contains: Live canvas editing (Pattern B), node color mapping
- Key files: `canvas-live-editor.ts` (probes for and uses undocumented internal API)

**`src/sessions/`:**
- Purpose: Persist in-progress protocol sessions to disk for resume-on-reopen
- Contains: Session data model, CRUD service with node ID validation
- Key files: `session-service.ts` (save/load/clear with WriteMutex)

**`src/snippets/`:**
- Purpose: Manage reusable text templates (JSON with placeholders, Markdown verbatim)
- Contains: Snippet types, full CRUD service with path safety, canvas reference sync
- Key files: `snippet-service.ts` (listing, loading, saving, renaming, moving), `canvas-ref-sync.ts` (vault-wide ref rewrite)

**`src/views/`:**
- Purpose: All UI code -- Obsidian ItemView subclasses, modals, widgets
- Contains: 3 main views (runner, editor, snippet manager), 7 modals, 2 widgets
- Key files: `runner-view.ts` (main user-facing view), `editor-panel-view.ts` (node authoring), `snippet-manager-view.ts` (library tree)

**`src/styles/`:**
- Purpose: Per-feature CSS files concatenated by esbuild into root `styles.css`
- Contains: One CSS file per feature area
- Constraint: New features add a new file here AND register it in `esbuild.config.mjs` CSS_FILES array
- Constraint: Append-only per phase -- never rewrite existing sections

**`src/utils/`:**
- Purpose: Shared infrastructure helpers
- Contains: Folder creation helper, per-path async mutex
- Key files: `write-mutex.ts` (wraps async-mutex, used by all services that write to vault)

**`src/types/`:**
- Purpose: Ambient type declarations for undocumented Obsidian APIs
- Contains: `canvas-internal.d.ts` (CanvasViewInternal, CanvasInternal, CanvasData)

## Key File Locations

**Entry Points:**
- `src/main.ts`: Plugin class (`RadiProtocolPlugin extends Plugin`) -- registers views, commands, context menu, instantiates services
- `esbuild.config.mjs`: Build entry -- bundles `src/main.ts` -> `main.js`, concatenates CSS

**Configuration:**
- `src/settings.ts`: `RadiProtocolSettings` interface, `DEFAULT_SETTINGS`, `RadiProtocolSettingsTab`
- `tsconfig.json`: TypeScript compiler options (strict mode, ES2018 target)
- `esbuild.config.mjs`: Build pipeline, CSS_FILES order, dev vault copy plugin
- `manifest.json`: Obsidian plugin metadata (id, name, version, minAppVersion)

**Core Logic:**
- `src/graph/canvas-parser.ts`: Parses canvas JSON string into `ProtocolGraph`
- `src/runner/protocol-runner.ts`: Protocol execution state machine (start, advance, undo, loops, snippets)
- `src/snippets/snippet-service.ts`: Full CRUD for snippets with path safety and mutex
- `src/sessions/session-service.ts`: Session persistence CRUD
- `src/canvas/canvas-live-editor.ts`: Live canvas editing via internal API

**Testing:**
- `src/__tests__/`: Unit test files
- `src/__tests__/fixtures/`: Test fixture data (canvas JSON, snippet JSON/MD files)
- `src/__tests__/test-utils/`: Shared test helpers
- `src/__mocks__/`: Manual mocks (e.g., Obsidian module mock)

## Naming Conventions

**Files:**
- kebab-case for all TypeScript source files: `canvas-parser.ts`, `runner-state.ts`
- kebab-case for CSS files: `runner-view.css`, `snippet-manager.css`
- Type declaration files: `*.d.ts` in `src/types/`
- Test files: `*.test.ts` in `src/__tests__/`

**Directories:**
- lowercase kebab-case: `graph/`, `runner/`, `views/`, `canvas/`
- Double-underscore prefix for test infrastructure: `__tests__/`, `__mocks__/`

**Exports:**
- Classes: PascalCase (`CanvasParser`, `ProtocolRunner`, `RunnerView`)
- Types/Interfaces: PascalCase (`RPNode`, `RunnerState`, `PersistedSession`)
- Constants: UPPER_SNAKE_CASE for view type strings (`RUNNER_VIEW_TYPE`), PascalCase for maps (`NODE_COLOR_MAP`)
- Functions: camelCase (`renderSnippet`, `ensureFolderPath`, `validateSessionNodeIds`)

## Where to Add New Code

**New Node Type:**
1. Add variant to `RPNodeKind` union in `src/graph/graph-model.ts`
2. Define interface (e.g., `NewTypeNode extends RPNodeBase`) in `src/graph/graph-model.ts`
3. Add to `RPNode` union in `src/graph/graph-model.ts`
4. Add parsing case in `src/graph/canvas-parser.ts` `parseNode()` switch
5. Add traversal case in `src/runner/protocol-runner.ts` `advanceThrough()` switch
6. Add color mapping in `src/canvas/node-color-map.ts`
7. Add editor form case in `src/views/editor-panel-view.ts` `buildKindForm()` switch
8. Add runner UI case in `src/views/runner-view.ts` `render()` switch
9. Add option to node type dropdown in `src/views/editor-panel-view.ts` `renderForm()`

**New View / Panel:**
1. Create view class in `src/views/` extending `ItemView`
2. Export view type constant (e.g., `MY_VIEW_TYPE`)
3. Register view in `src/main.ts` `onload()` via `this.registerView()`
4. Add activation method in `src/main.ts` (e.g., `activateMyView()`)
5. Add command in `src/main.ts` `onload()` via `this.addCommand()`

**New Modal:**
1. Create modal class in `src/views/` extending `Modal`
2. Use Promise-based result pattern (see `ResumeSessionModal`, `ConfirmModal` for examples)
3. Import and open from the calling view

**New CSS Feature:**
1. Create `src/styles/my-feature.css`
2. Add `'my-feature'` to `CSS_FILES` array in `esbuild.config.mjs`
3. Add CSS rules with phase comment: `/* Phase N: description */`
4. Run `npm run build` to regenerate `styles.css`

**New Service:**
1. Create service class in appropriate `src/` subdirectory
2. Accept `App` and/or `Settings` via constructor (dependency injection)
3. Instantiate in `src/main.ts` `onload()` and store as plugin property
4. Use `WriteMutex` for any vault write operations

**New Utility:**
1. Add to `src/utils/` if shared across multiple modules
2. Keep pure (accept Vault/App as parameters, not via import)

## Special Directories

**`.radiprotocol/` (in user's vault, not in source):**
- Purpose: Plugin data storage inside the Obsidian vault
- Contains: `snippets/` (JSON/MD snippet files), `sessions/` (session JSON files)
- Generated: Yes, at runtime
- Committed: No (lives in the user's vault, not the plugin repo)

**`node_modules/`:**
- Purpose: npm dependencies
- Generated: Yes, via `npm install`
- Committed: No

**Key dependency rules:**
- `src/graph/` and `src/runner/` have ZERO Obsidian imports (pure TypeScript)
- `src/views/` depends on everything else but nothing depends on views (leaf layer)
- Services accept `App` via constructor injection, not via global import
- `src/utils/` is a shared foundation with no domain logic
