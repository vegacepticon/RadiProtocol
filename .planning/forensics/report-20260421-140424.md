# Forensic Report — почему GSD команды жгут Max-лимит

## TL;DR

Проблема **не в коде проекта** и не в зависших циклах. Проблема в **раздутом контексте `.planning/`**, который GSD-команды и их субагенты загружают каждый раз. Базовая «стоимость запуска» команды — ~41k токенов на core planning-файлах ещё до первого анализа. К ним добавляется:

1. `.planning/STATE.md` **57,6 KB** (~14k токенов) — раздут per-plan executive log'ом с 1500-словными абзацами за каждый из 34 завершённых планов.
2. Все опциональные GSD workflow-агенты **включены** в `config.json`, включая `ai_integration_phase` для проекта, где AI **нет**.
3. Файлы PLAN/RESEARCH/PATTERNS — 25–55 KB каждый; один phase-researcher тянет 60–100 KB артефактов.
4. Признаки повторной работы: Phase 54 прошла **3 UAT-раунда** с фиксами, Phase 56 — это явный реверс решений Phase 51 (D-13/D-16), Phase 50.1 — вставка-патч после Phase 50.

## Anomalies Detected

### 1. STATE.md раздут per-plan execution log'ом — **HIGH**
**Evidence:**
- Раздел `### v1.8 Execution Log` в `STATE.md` содержит абзац-портянку на каждый из ~19 планов milestone v1.8, от ~800 до 1700 слов.
- Пример (49-03): абзац содержит grep-подсчёты (`chooseLoopBranch` = 1), списки deviations, хэши коммитов, количество строк JSDoc. Вся эта информация **уже есть** в `{phase}-{plan}-SUMMARY.md`.
- Файл целиком — 57,6 KB, ~14k токенов, грузится в каждый `/gsd-*` вызов через frontmatter+body.

**Interpretation:** STATE.md исторически задумывался как «курсор» (где мы, что дальше), но превратился в дубликат per-plan SUMMARY. Результат — **баннер ~14k токенов на каждую команду** + большую долю этого контента субагенты тоже видят.

**Cost impact:** ориентировочно **10–14% от стоимости каждой команды** уходит на перечитывание того, что уже закоммичено в SUMMARY.md-файлах и `git log`.

### 2. `ai_integration_phase: true` на проекте без AI — **HIGH**
**Evidence:**
- Проект — Obsidian-плагин для радиологов (рендер протоколов из canvas-графа). Никакого LLM/ML/embedding.
- `config.json`: `ai_integration_phase: true` оставлено по умолчанию.
- Если любая фаза отмечена как AI-relevant, `/gsd-ai-integration-phase` спавнит **3 агента параллельно**: framework-selector, domain-researcher, eval-planner (+ ai-researcher). Каждый — с веб-поиском и context7 MCP. Это 40–80k токенов впустую.

**Interpretation:** Флаг заведён «на всякий случай», никогда не стрелял полезно, но если случайно сработает на фазе — грохнет четверть лимита за минуту.

### 3. Все workflow-гейты включены + `mode: yolo` — **MEDIUM**
**Evidence:**
- `mode: yolo` по задумке GSD = минимум user-interrogation, быстрый проход.
- Но `research: true` + `plan_check: true` + `verifier: true` + `nyquist_validation: true` + `ui_phase: true` + `ui_safety_gate: true` означает: **6 последовательных агент-ранов на каждую фазу**, каждый тащит STATE+ROADMAP+phase artifacts.
- Для CSS-only плана (например, 47-03 UAT revision) это overkill — запускается тот же пайплайн, что для реального реактивного сервиса.

**Interpretation:** Конфиг противоречит сам себе. Для мелких/механических плановых фаз нужен light-профиль.

**Cost impact:** каждый опциональный агент в plan-phase ~= 15–30k токенов. Отключение 2–3 из них экономит **30–80k на фазу**.

### 4. Per-plan file size bloat — **MEDIUM**
**Evidence:**
- Plans 40–55 KB — это норма в этом проекте (52-01 = 56 KB, 48-01 = 44 KB).
- Для сравнения: сам `src/views/runner-view.ts` (по ссылкам из SUMMARY) — имеет порядка 700-800 строк. План, который говорит «добавить метод skip() в ProtocolRunner», — 25 KB текста.
- RESEARCH.md 44–68 KB — часто содержит markdown-копии фрагментов кода, которые планировщик перечитывает вместо исходников.

**Interpretation:** Планы пишутся «обложно» — с кодом, гипотетическими диффами, зеркальными ссылками. Executor их читает, researcher уже их прочитал, plan-checker читает их снова.

### 5. High rework rate — переоткрытые решения — **MEDIUM**
**Evidence:**
- Phase 48.1 вставлена сразу после 48 (UAT follow-up cosmetic).
- Phase 50.1 вставлена после 50 — явно «supersedes Phase 49 D-07 alias» (конфликт решений).
- Phase 56 полностью реверсит Phase 51 D-13/D-16 (`ROADMAP §Evolution`: «overturns Phase 51 D-13 (auto-insert on single-edge file-bound Snippet) and D-16 per user verdict during Phase 51 UAT»).
- Phase 54 потребовала **3 раунда UAT-фиксов** (`e3e8cb1` 5 fixes → `22e7b0b` round 2 → `f4c2352` round 2b) + отдельный code-review cycle (`cd2baa3` CR-01..WR-05).

**Interpretation:** Processed locked-decisions слишком рано, до того как реальное поведение проверено в UAT. Каждый overturn = новая phase + ещё один полный plan-execute цикл.

### 6. Fragmented plan granularity — **LOW/MEDIUM**
**Evidence:**
- Phase 51 — 6 планов. Phase 55 — 4 плана. Phase 54 — 4 плана.
- Каждый план = отдельный plan-phase вызов или orchestrator-step, со своим plan-check + summary + commit.

**Interpretation:** Микрогранулярность даёт много SUMMARY-файлов (backward-log rot) и multiplicates plan-check passes. Для Phase типа «CSS tweak» 1 plan достаточно.

### 7. Uncommitted planning docs (не ошибка, но шум) — **LOW**
**Evidence:** `git status` показывает `51-HUMAN-UAT.md`, `53-PATTERNS.md`, `54-CONTEXT.md`, `54-REVIEW.md` неcommitted.

**Interpretation:** Не влияет на стоимость, но следующие команды будут видеть их как untracked каждый раз.

## Root Cause Hypothesis

**Основная причина высокой стоимости plan-phase и execute-phase — не сам процесс, а накопленная в `.planning/` нагрузка**, которую каждый вызов загружает. Три фактора перемножаются:

1. **Большой baseline** (STATE 57 KB + ROADMAP 48 KB = 25k токенов загружаемых *всегда*).
2. **Полный набор optional-агентов активен** (6 агентов × 20–40k контекста каждый).
3. **Большие per-phase артефакты** (PLAN 25–55 KB × несколько планов за фазу).

При балансированном профиле + web-поиске в research-агентах одна plan-phase легко набирает **300–500k токенов** суммарных чтений (а на большой фазе типа 52 — до 700k). Плюс 15–20 commits с git-ops. На 5-часовом Max-лимите это действительно ~30–50%.

Дополнительный множитель — **rework**: Phases 48.1, 50.1, 56 — это «лишние» фазы, которые можно было бы избежать более ранним UAT-probing или менее жёсткой фиксацией decisions до фактической проверки. Каждая URGENT-вставка = ещё один полный цикл plan/execute.

## Recommended Actions

### Немедленные (экономят ~30–50% cost за одну фазу)

1. **Обрезать `STATE.md § v1.8 Execution Log`.** Заменить per-plan абзацы на одну строку формата `Phase 49 Plan 03 (98df8ee): runner dispatch via isExitEdge. See 49-03-SUMMARY.md` — вся детализация уже есть в SUMMARY.md и `git log`. Ожидаемый размер после: ~15 KB (в 4 раза меньше, -10k токенов с каждого запуска).

2. **Выключить `ai_integration_phase`** в `.planning/config.json`:
   ```json
   "ai_integration_phase": false
   ```
   Проект не содержит AI. Если когда-нибудь начнётся — включить снова.

3. **Выключить `ui_phase` / `ui_safety_gate`** для CSS-тривиальных фаз. Альтернатива — держать включённым, но переводить на `--quick` или пропускать для фаз с тегом `ui-css-only`. Обсидиан-плагин без сложного UI это не оправдывает 2 агента на фазу.

### Среднесрочные

4. **Писать менее объёмные PLAN.md.** Целевой размер — 10–15 KB, не 40–55. Plan не должен дублировать код; он должен ссылаться на файлы и указывать точки изменения. Код-снипеты в PLAN — самая «жирная» часть.

5. **Архивировать завершённые фазы.** Есть команда `/gsd-cleanup`. 12 активных phase-директорий с суммарно 2,35 MB артефактов — кандидаты на архив в `.planning/milestones/v1.8-phases/` после `/gsd-complete-milestone`.

6. **Уменьшить гранулярность планов на простых фазах.** Phase с 6 планами означает 6 plan-check passes + 6 executor spawns. Где фаза тривиально атомарна — 1–2 плана.

7. **Probe ключевые UX-решения через быстрый prototype/sketch перед тем как lock'ить в CONTEXT.md.** Phase 51 зафиксировала D-13 «auto-insert on file-bound Snippet», потом Phase 56 полностью реверсила. Одна сессия `/gsd-sketch` или `/gsd-spike` до locked-decisions сэкономила бы всю Phase 56.

### Долгосрочные

8. **Перевести `model_profile` в `budget` для repetitive-агентов** (verifier, nyquist-auditor) — они делают грепы, им Opus не нужен. Оставить `quality` для phase-planner и code-reviewer. Настраивается через `/gsd-set-profile`.

9. **Настроить `.planning/STATE.md` retention policy** — держать «последние 3 фазы» в экзекьютив-логе, старее переносить в `.planning/milestones/v1.X-LEARNINGS.md`.
